# Brief: Iterative Critique Loop for /simulate (D22)

Generated by /think refine on 2026-04-15

## Problem Statement
V2 sim pipeline scores 3.70/5 vs eval_intake's 4.73/5. The cross-model consensus (D22) is that the gap is structural — eval_intake's quality came from 17 rounds of watching-and-fixing, not from architecture. Automation can't replace iteration. The fix is a developer-in-the-loop critique loop: run a sim, review the transcript, annotate where it failed, feed annotations back, rerun.

## Proposed Approach
Freetext annotation on markdown transcripts. Zero new abstractions — layer a `--critique` flag onto the existing sim_pipeline.py.

**The loop:**
1. `sim_pipeline.py --skill explore` → produces transcript markdown in output dir
2. Developer reads transcript, writes inline annotations (freetext comments next to turns)
3. `sim_pipeline.py --skill explore --critique path/to/annotated-transcript.md` → system reads annotations, injects them as additional executor context for the next run
4. Compare judge scores before/after. Repeat until satisfied.

**How annotations become context:** The annotated transcript is fed to an LLM call that extracts adjustment directives (e.g., "push harder on hidden truth at turn 3", "match the user's casual register"). These directives are injected as a system reminder block in the executor's prompt for the next run — same mechanism as existing turn_hooks but seeded from developer feedback instead of hardcoded rules.

**What stays the same:** sim_driver.py (3-agent loop), sim_compiler.py (IR extraction), persona loading, rubric, judge scoring. No changes to the core simulation machinery.

## Key Assumptions
1. Freetext annotations carry enough signal for an LLM to extract actionable directives — structured tags aren't needed for the MVP
2. One round of annotation meaningfully moves judge scores (≥0.5 on at least one dimension)
3. Existing sim infrastructure (driver, compiler, personas, judge) is reusable as-is for the "run" step
4. Developer annotation time is acceptable (~5-10 min to read a transcript and comment)
5. The executor can act on injected directives without conflicting with the skill procedure

## Risks
1. **Freetext ambiguity** — LLM misinterprets annotations and makes wrong adjustments. Mitigation: show the extracted directives to the developer before rerunning (confirmation step).
2. **Symptom-targeting** — Developer annotates "push harder" when the real fix is the persona card format. Mitigation: if 2 rounds of annotation don't improve a dimension, surface the underlying input (persona/rubric) as the probable root cause.
3. **Transcript length** — 15-turn transcripts are painful to read. Mitigation: emit a judge score summary at the top with per-dimension scores, so developer can focus annotation on failing dimensions.
4. **Directive conflicts** — Injected directives may contradict the skill procedure. Mitigation: directives are framed as "in addition to your procedure" not "instead of."

## Simplest Testable Version
**Scope:** /explore skill only (has anchor personas, rubric, baseline scores from spike).

**New code:**
- `--critique <path>` flag on sim_pipeline.py
- `extract_directives(annotated_transcript) -> list[str]` function (~30 lines) — LLM call that reads the annotated transcript and outputs structured adjustment directives
- `critique_hook(directives)` — a turn_hook factory that injects directives as a system reminder at turn 1

**Test protocol:**
1. Run baseline: `sim_pipeline.py --skill explore --persona-source anchor --rubric-source fixtures/.../rubric.json`
2. Pick the lowest-scoring run. Export transcript as markdown.
3. Developer annotates the transcript (freetext inline comments).
4. Rerun with critique: `sim_pipeline.py --skill explore --critique annotated.md --persona-source anchor --rubric-source fixtures/.../rubric.json`
5. Compare judge scores.

**Success criteria:** ≥0.5 improvement on at least one dimension after one annotation round. If hidden_truth (the hardest dimension, 80% failure rate) improves, strong signal.

**Estimated new code:** ~80-120 lines in sim_pipeline.py + a small transcript formatter. No new files, no new abstractions.
