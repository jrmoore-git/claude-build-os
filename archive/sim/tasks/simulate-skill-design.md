# Design: /simulate — Zero-Config Skill Simulation

Generated by /think discover on 2026-04-14
Branch: main
Status: DRAFT
Mode: Builder

## Problem Statement
BuildOS has 21 skills and a cross-model debate engine. When skills are written or modified, there's no way to validate them without real usage. Over 72+ hours, the team ran 10+ ad-hoc simulations — persona tests, smoke-tests, delta proofs, cross-model consensus loops — all manually orchestrated each time. eval_intake.py exists but is hardcoded to explore intake. The default recommendation ("just use it on real work") defers validation indefinitely and makes the user the bug-finder.

## What Makes This Cool
Say `/simulate /elevate` and Claude reads the SKILL.md, auto-generates a realistic scenario, picks model roles, runs it end-to-end, scores it, and tells you what broke — zero configuration. Time compression: experience emergent behavior at 100x speed instead of waiting for organic usage to surface bugs.

## Five Simulation Modes (from evidence)

### 1. Smoke-test: "Does it work?" [V1]
Read the SKILL.md. Extract fenced bash code blocks and referenced file paths. Run each command. Report what passes and what fails.
- **Input:** Target skill path
- **Output:** Pass/fail per command, missing files, broken paths, with PASS/FAIL/SKIPPED status per item
- **Models:** Single agent, no cross-model needed
- **Safety:** Read-only by default. Skip commands containing `rm`, `git push`, `git reset`, or write redirects to non-tmp paths. Flag skipped commands in report.
- **External APIs:** Real calls (the point is: does it work?). Flag API failures distinctly from logic failures.
- **Example:** Today's /elevate audit — found mktemp bug on BSD

### 2. Quality eval: "Is the output good?" [V1]
Generate a realistic user scenario. One agent follows the SKILL.md procedure, another plays the user. A third judges the output.
- **Input:** Target skill path + optional scenario hint
- **Output:** Transcript + judge scores (5 dimensions) + issues found
- **Models:** Skill-runner (session model), simulated user (different model via debate.py), judge (third model via debate.py)
- **External APIs:** Skill-runner agent makes its own tool calls naturally. External API results are real if available, degraded if not — same as real usage.
- **Example:** Intake 9-persona test — Claude interviews, Gemini plays persona, GPT judges

### 3. Adversarial: "Can it be broken?" [V2]
Generate edge-case scenarios designed to break the skill: contradictory inputs, bail signals, hostile users, empty inputs, malformed data.
- **Input:** Target skill path
- **Output:** Failure modes found, structural fixes needed
- **Models:** Skill-runner + adversarial user agent
- **Example:** Intake pass 3 — contradictions, skips, solution-seekers

### 4. Delta proof: "Does it matter?" [V3]
Run the same scenario twice — once with the skill/feature, once without. Blind judge scores both. Measures actual impact.
- **Input:** Target skill path + baseline (auto: run without skill instructions; manual: user-supplied baseline)
- **Output:** Before/after scores, delta per dimension
- **Models:** Two runs + blind judge
- **Example:** Intake with vs without — 2.0→4.7/5

### 5. Consensus: "Do models agree?" [V3]
Run the same evaluation with 3 different models independently. Compare scores. Consensus = real signal; disagreement = taste difference.
- **Input:** Target skill path + evaluation criteria
- **Output:** Per-model scores, convergence analysis, variance report
- **Models:** 3 independent judges. Degraded: 2 models = still useful comparison; 1 = skip with explanation.
- **Stopping rule:** If 3/3 agree within 0.5 points, stop. If spread > 1.0, report disagreement areas.
- **Example:** Register/flow convergence rounds 12-17

## Build Phases
- **V1:** Smoke-test + quality eval (covers 80% of evidence instances — 8 of 10)
- **V2:** Adversarial (adds edge-case discovery)
- **V3:** Delta proof + consensus (adds impact measurement and cross-model agreement)

Each phase ships when the previous is validated via self-simulation.

## Premises
1. A SKILL.md contains enough structure to auto-generate a realistic test scenario without human input.
2. The same skill can handle both infrastructure smoke-tests and interactive quality evals.
3. Zero-config is the right default — custom personas/rubrics are premature for this project's scale.
4. A skill (not a script) is the right implementation — Claude reasoning about what to test beats a Python parser.
5. All five modes are needed (evidence from 10 real instances maps to all 5), but phased delivery (V1-V3).

**Known limitation (Premise 1):** Some SKILL.md files may be too sparse for auto-scenario generation. Fallback: if confidence is low, generate 2-3 candidate scenarios and pick the most exercising one, or ask user for a one-line scenario hint.

**Known limitation (Premise 4):** Tool-heavy skills (those depending on specific tool behavior rather than prompt-following) may have lower simulation fidelity. The simulation tests the skill's reasoning and procedure, not exact tool integration. Flag this in reports as "simulation fidelity: prompt-only" vs "simulation fidelity: tool-backed."

## Independent Perspective
Reviewer (PM persona via debate.py) suggested:
- **Chaos Monkey auto-fix loop** — find bug → auto-rewrite skill → re-verify → present diff. Rejected: violates "LLM may decide, software must act" — but "find → suggest fix → human approves" variant is worth considering post-V3.
- **Promptfoo integration** — external eval engine. Rejected: new dependency (security review), can't handle interactive turn loops, overkill for smoke-tests.
- **AskUserQuestion interception** — key design insight adopted: simulation doesn't invoke the skill directly. An agent reads the SKILL.md procedure and follows it, with another agent as user. AskUserQuestion becomes agent-to-agent message exchange.

## Recommended Approach
**Approach A: Skill-reads-skill.** SKILL.md procedure. Claude reads the target skill, reasons about what to test, spawns agents, uses debate.py review for judging. No new Python scripts in V1 (eval_intake.py stays for deep intake-specific evals).

### How it works
1. User invokes `/simulate <target-skill> [--mode MODE]`
2. Skill reads `.claude/skills/<target>/SKILL.md`
3. Validates target exists and has parseable structure. If not found, list available skills.
4. Selects mode (precedence: explicit `--mode` > user intent cues > SKILL.md heuristics > default smoke-test):
   - Has bash code blocks → default: smoke-test
   - Has AskUserQuestion references → default: quality eval
   - User says "break it" → adversarial
   - User says "does it matter" → delta proof
   - User says "do models agree" → consensus
5. For smoke-test: spawn single agent to extract and run fenced bash blocks, check file paths
6. For quality eval: spawn agent to simulate a user scenario, run target skill procedure, judge output via debate.py review
7. Output structured report

### Interactive skill handling
The simulation never invokes the target skill via the Skill tool. Instead:
- The simulation agent reads the SKILL.md and follows its procedure step by step
- For quality eval, a simulated user (via debate.py review or Agent with different model) provides responses
- When the procedure would call AskUserQuestion, the simulation generates the response based on the scenario
- This works because skills are prompt documents (D3) — an agent following the procedure IS the skill running
- **Fidelity caveat:** Tool-heavy skills (those where tool side-effects matter more than prompt logic) get lower fidelity simulation. Report includes fidelity tier: prompt-only, tool-using, or external-dependent.

### Scenario generation
For quality eval mode, the skill auto-generates a scenario:
1. Read the target SKILL.md description, frontmatter, and procedure
2. Identify what kind of input the skill expects (problem statement, code diff, document, etc.)
3. Generate a realistic scenario that exercises the skill's main path
4. If confidence is low (sparse SKILL.md), generate 2-3 candidates and pick the most exercising one

### Scoring rubric (minimum schema)
For quality eval, the judge scores on 5 dimensions (1-5 each):
1. **Task completion** — did the skill accomplish its stated purpose?
2. **Instruction adherence** — did it follow the SKILL.md procedure?
3. **Output quality** — is the output well-structured, actionable, and appropriate?
4. **Robustness** — did it handle ambiguity, missing info, or edge cases gracefully?
5. **User experience** — was the interaction natural, appropriately paced, and respectful of the user's time?

Pass criteria: task completion must be 3+ (blocking gate — a skill that doesn't complete its task fails regardless of other scores), all other dimensions 3+, average 3.5+. These are intentionally lower than eval_intake.py's 4.5+ bar — this is a smoke-level quality check, not a production readiness gate.

### Failure handling
Result statuses: PASS, FAIL, INCONCLUSIVE (ambiguous result), SKIPPED (command unsafe or API unavailable).
Each status includes cause, evidence, and retry guidance.

## Resolved Questions (from spec review)
1. Simulated user model: configurable, default Gemini (via debate.py persona routing).
2. External APIs: smoke-test = real calls; quality eval = natural (agent makes tool calls as it would in real usage).
3. Output: `tasks/<skill-name>-simulate.md` (consistent with other task artifacts).
4. Auto-run: manual only in V1. Hook-triggered option deferred to post-V1.

## Success Criteria
- Can run `/simulate /elevate` and get a useful report without configuration
- Smoke-test mode catches the mktemp bug we found today (ground truth validation)
- Quality eval mode produces transcript + judge scores for an interactive skill
- Target runtime: <3 min for smoke-test, <10 min for quality eval (not hard gates — report partial results if exceeded)
