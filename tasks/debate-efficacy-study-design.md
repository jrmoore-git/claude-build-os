# Design: Debate Efficacy Study

Generated by /think discover on 2026-04-17 (autopilot)
Branch: main
Mode: Builder (internal methodology research)

## Problem Statement

The cross-model debate system (`scripts/debate.py`, `/challenge`, `/review`) is load-bearing governance infrastructure for BuildOS — it gates planning (Stage 1), refines architecture (Stage 2), and reviews implementation (Stage 3). Its design is grounded in theory + localized paired validation (frame-lens n=5), but we have **no direct comparison against cheaper alternatives**. Specifically: we don't know whether multi-model adds value beyond multi-persona, which is the decision-relevant question because dropping cross-model would cut ~10x of debate API spend.

Existing evidence is mixed:
- **Pro:** Frame-lens-validation (n=5) showed panel-of-4 > panel-of-3 with ~30 novel MATERIAL findings and one verdict flip.
- **Contra:** D5 (single model + divergence prompts ≥ multi-model for thinking modes), L28 (3 independent models all unanimous-and-wrong when operational evidence missing), L33 (`--enable-tools` produced 0.98-confidence false claim a `ls hooks/` would have refuted).

None of these compare debate to `/review`-with-just-Claude-subagents. That's the gap.

## What Makes This Worth Doing (Builder Mode)

This is the next-session action already named in `docs/current-state.md`: "Run paired output-quality audit (n=5, paired comparisons, quality first per L44) across other personas and other multi-model systems." The study formalizes that work into a controlled comparison.

Decision-relevant outcomes:
- **If D > C substantially:** cross-model is validated; keep current spend.
- **If C ≈ D:** drop cross-model (~10x cost reduction on `/challenge` and `/review`), keep persona structure.
- **If C > D:** cross-model is actively harmful (correlation or quorum bias); bigger rethink.
- **If B ≈ C ≈ D:** most of our review infrastructure is theater; simplify to single-call review + persona prompts.

Any of these outcomes changes framework behavior. That makes the study load-bearing, not academic.

## Premises

1. **Multi-model vs multi-persona is the critical isolated variable.** The most decision-relevant comparison is arm C (Claude-only persona panel) vs arm D (current full debate). If they're equivalent, the model diversity is decoration.
2. **Ground truth is accessible but expensive.** Past proposals have outcomes (shipped / abandoned / shipped-then-reworked) visible in git + lessons.md. Canonicalizing findings requires manual labeling — no silver bullet.
3. **Retrospective contamination is real.** Every arm run at t=now has seen the proposal's eventual outcome in training or context. This biases findings toward hindsight-match. Control (external judge, dual-label) reduces bias but doesn't eliminate it.
4. **L44 sets the evidence bar.** Any tuning decision following from this study must clear n≥3 (ideally 5+) paired comparisons on output quality, not speed/cost.
5. **The study must not use debate.py to evaluate debate.py.** Judge model (Gemini) must be held out from the arms being compared; otherwise contamination loops.
6. **`stores/debate-log.jsonl` does not record outcomes.** 447 lines of phase/debate_id/mapping — no ground-truth labels. Labeling is net-new manual work.

## Approaches Considered

### Approach A: Full 4-arm study, retrospective + prospective (originally scoped)

**Summary:** n=15 retrospective proposals + n=5 held-out prospective. Run arms A (no review), B (single Claude subagent), C (Claude-only persona panel), D (full debate). External Gemini judge + 20% dual-label with Justin.

**Effort:** L (~2-4 weeks elapsed for prospective arm)
**Risk:** Med (labeling subjectivity, prospective arm requires new proposals to arrive)
**API cost estimate:** $300-800 across all runs.
**Pros:**
- Directly answers "is debate worth anything vs alternatives"
- Prospective arm controls contamination
- High decision power across all pairwise comparisons
**Cons:**
- Slow (prospective arm blocks final answer ~1 month)
- Cost + labeling effort
- n=5 prospective is still small for 4-way comparison
**Reuses:** frame-lens-validation paired-comparison methodology; debate.py dispatch; existing proposals on disk.

### Approach B: Retrospective-only, 4-arm (strip prospective)

**Summary:** Same as A, drop prospective arm. n=15-20 retro proposals, all 4 arms, external judge.

**Effort:** M (~1 week)
**Risk:** Med-High (contamination unmitigated; results suggestive not definitive)
**API cost estimate:** $150-400
**Pros:**
- Ships fast; all data exists now
- Still covers the decision-relevant C-vs-D question
**Cons:**
- Contamination is the real limit — any conclusion carries "but Claude saw this proposal's outcome in context" caveat
- Doesn't unlock prospective methodology for future studies
**Reuses:** same as A.

### Approach C: Minimal isolation — C vs D only, n=5 frame-lens proposals (RECOMMENDED)

**Summary:** Run arm C (Claude-only persona panel — architect, security, pm, frame-structural, frame-factual, all as Claude Opus subagents) vs arm D (current multi-model debate) on the same 5 proposals used in frame-lens-validation: autobuild, explore-intake, learning-velocity, streamline-rules, litellm-fallback. Use frame-lens-validation's already-canonicalized findings as ground truth. Compare per-finding precision and novel-finding count. One-day exercise.

**Effort:** S (~1 day work, ~$20-40 API)
**Risk:** Low
**Pros:**
- Answers the one decision-relevant question fastest: does cross-model add value beyond multi-persona?
- Ground truth already exists (frame-lens-validation.md documents what was caught; git commits document what mattered)
- Matches L44 n=5 evidence bar exactly
- Zero new infrastructure
- "Simplest version wins unless there's evidence otherwise" (CLAUDE.md override rule)
**Cons:**
- Doesn't answer "is debate worth anything vs no review" — only marginal value of cross-model
- n=5 is low power for detecting small effects
- Uses proposals debate.py has already been run against (weaker than held-out)
- No arm A (no review) or arm B (single-shot single-model) baselines
**Reuses:** frame-lens-validation canonical findings, 5 proposals, debate.py infrastructure.

### Approach D: Instrument-and-wait (prospective-only)

**Summary:** Randomize arm assignment in future `/challenge` and `/review` invocations; capture outcome as it accrues. Wait 3-6 months for sufficient n.

**Effort:** S initial, L elapsed
**Risk:** Low
**Pros:** Zero contamination; natural randomization; cheap per-event
**Cons:** Can't answer now; requires 3-6 month patience; requires instrumentation to `debate-log.jsonl`

## Recommended Approach: C, with B as natural escalation if C is inconclusive

**Why C first:**

The simplicity override rule applies here. Approach C is the smallest version that answers the decision-relevant question. If C shows clear separation between arms, we have our answer at ~$20 cost in ~1 day. If C is inconclusive, escalate to B (retro-only 4-arm) to get broader coverage. A is overkill for the marginal-value question and should only run if B is also inconclusive.

**Concretely, Approach C design:**

1. **Arms (2):**
   - **C (Claude-only panel):** 4 parallel Claude Opus subagents via the `Agent` tool, each with one of: architect prompt, security prompt, pm prompt, frame-dual prompt. Same prompts as current `debate.py` personas; model swapped to claude-opus-4-7 across the board. No LiteLLM dispatch — directly via Agent subagents.
   - **D (current debate):** `scripts/debate.py challenge` with default persona list and `--enable-tools` to match frame-lens-validation conditions.

2. **Sample:** The 5 frame-lens-validation proposals (autobuild, explore-intake, learning-velocity, streamline-rules, litellm-fallback) — ground truth already documented, matches prior methodology.

3. **Metrics:**
   - **Per-finding precision:** external Gemini 3.1 Pro judges each MATERIAL finding against ground truth (was this a real issue per the proposal's actual outcome?)
   - **Recall against canonical list:** for each proposal, % of frame-lens-validation's documented MATERIAL findings that arm C surfaced
   - **Unique-to-arm findings:** novel MATERIAL findings that arm C caught but arm D missed, and vice versa
   - **Cost per finding:** (API spend + wall time) / MATERIAL findings

4. **Contamination controls:**
   - External judge is Gemini 3.1 Pro (not part of arm C, not part of arm D's default rotation for the personas being compared — overlap is tolerable but tracked)
   - 20% dual-label by Justin (1 of 5 proposals) to measure judge agreement
   - Record arm configuration in `stores/debate-log.jsonl` extensions so future analysis can distinguish

5. **Output:** `tasks/debate-efficacy-study-c.md` with raw findings matrix, judge scores, cost comparison, conclusion paragraph. If findings recommend action, a follow-up `/challenge` gates the action.

**Escalation rule:** If the C-vs-D distinction is smaller than the paired-comparison noise floor (roughly: no arm catches ≥2 more unique MATERIAL findings than the other on 3+ of 5 proposals), escalate to Approach B. If Approach B still inconclusive, run A.

## Open Questions

1. **How do we handle arm C's tool access?** Arm D (current debate) uses `--enable-tools` for frame-factual. For parity, should arm C also get tool access via Agent subagents? Leaning yes — the comparison should be persona-structure-identical except model.
2. **Is Opus-only arm C a fair comparison to multi-model arm D?** Opus is the strongest model in the rotation. If we use Opus-only for C, we're giving arm C the best single model. Alternative: rotate Claude model across personas (Opus architect, Sonnet pm, Haiku security, Opus frame) to isolate "no cross-family" rather than "no model diversity at all." This is the harder question.
3. **Does arm D get tool access if arm C doesn't?** Or do we run two C variants (tools-on, tools-off) to match frame-lens-validation Round 1 structure? Leaning: match current production `/challenge` defaults for both — that's what we're deciding about.
4. **How do we handle LiteLLM rate-limit interactions?** Arm D goes through LiteLLM; arm C goes direct via Claude Code. Rate-limit experience differs. Not a study concern but worth noting for result interpretation.

## Success Criteria

- [ ] Canonical MATERIAL finding list compiled for each of the 5 proposals (combined from frame-lens-validation.md + git-history review)
- [ ] Arm C run with Agent subagents for each of the 5 proposals, findings logged
- [ ] Arm D output already on disk (from frame-lens-validation runs) — verify retrievable
- [ ] External judge (Gemini 3.1 Pro) scores each MATERIAL finding against canonical list; agreement rate measured on 20% dual-label
- [ ] Per-arm precision, recall, unique-catches, cost/finding computed
- [ ] Study writeup clearly states: keep cross-model / drop cross-model / inconclusive-escalate-to-B

## Known Limits (from conversation)

- **Long-tail consequences:** a missed finding that causes an incident 6 months out won't show up in this study. Point-estimate only.
- **Reviewer fatigue effects:** single-run measurement doesn't capture how humans interact with debate output across many invocations.
- **Upstream shaping of thinking:** debate arguably influences *what gets proposed* in the first place (authors pre-empt challenges). This study can't measure that.
- **Small-n statistical power:** n=5 detects large effects reliably; can miss medium-size effects. This is the L44 threshold, not a statistical floor.
- **Contamination:** we've already run debate.py on these proposals. Any arm Claude runs at t=now has implicit memory of prior runs via training or session context.
