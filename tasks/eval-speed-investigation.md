## Hypotheses Tested (with panel results)

### H1: Serial `review` calls instead of parallel `review-panel` -- PRIMARY CAUSE

**Panel verdict: CONFIRMED (5/5, 4/5, 5/5)**

Evidence: E3 is dispositive. 37 serial `review` calls consumed 47.8 minutes. `review-panel` would parallelize each 3-model triplet, collapsing ~12 rounds from 47.8 min to ~16 min (3x speedup on the parallel portion). The code path difference is structural: `review` = 1 serial API call, `review-panel` = N parallel calls via ThreadPoolExecutor.

Panel noted: Missing explanation of WHY `review` was used instead of `review-panel`. Likely a session-level orchestration choice (the agent called `debate.py review` 3 times per round instead of `debate.py review-panel` once).

### H2: Persona sims orchestrated serially instead of using `--run-all` -- SECONDARY

**Panel verdict: POSSIBLE (3/5) to LIKELY (4/5)**

Evidence: E4 confirms `eval_intake.py --run-all` parallelizes with max_workers=3. The claim says "3 run serially" which implies the session ran individual `--persona` invocations. But `eval_intake.py` doesn't log to `debate-log.jsonl`, so there's no direct timing evidence. Impact: moderate (~5-10 min).

Panel noted: Shell history or orchestrator logs needed to confirm.

### H3: Claim's math is inaccurate -- FACTUAL ERRORS IN L19 AND D13

**Panel verdict: CONFIRMED (5/5, 5/5, 5/5)**

| Claim component | Stated | Actual |
|----------------|--------|--------|
| Rounds | 6 | 12+ (37 calls / 3 models) |
| Models per round | 3 (described as refine) | 3 (but these are `review` calls, not `refine`) |
| Per-call duration | 2-4 min | 1.3 min average |
| Eval portion | ~60 min | 47.8 min |
| Mechanism | "6 rounds of refine with 3 serial model calls each" | 12+ rounds of 3 serial `review` calls each |

The claim conflates `refine` (inherently serial, 1 model per round, correct by design) with the serial `review` evaluation loop (3 models per round, parallelizable). D13's root-cause statement ("18 serial API calls (6 rounds x 3 models)") propagates this conflation.

## Root Cause

**The 75-min wall-clock is approximately correct but the decomposition is wrong.**

The actual breakdown:
1. **~48 min**: 37 serial `review` calls across 12+ evaluation rounds (not 6 rounds of `refine`). Each round used 3 serial `review` calls when `review-panel` would have parallelized them.
2. **~15-25 min**: Persona sims + overhead (not directly measurable from logs, but plausible given 5 sims at ~3-5 min each).
3. **Per-call latency**: 77s average (1.3 min), not 2-4 min as claimed.

The fix identified in D13 (use `review-panel` instead of serial `review`) is correct, regardless of the math errors. Estimated savings: ~30 min on the review block alone.
