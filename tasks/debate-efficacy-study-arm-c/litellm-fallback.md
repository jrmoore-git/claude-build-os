# litellm-fallback — Challenger Reviews

## Challenger A (architect) — Challenges
Critical finding: **Anthropic fallback explicitly does not support tools** (line 737-738). This directly collides with the proposal — the repo has `debate_tools.py` with `llm_tool_loop` used for claim verification, and `frame_factual_model` uses "tools-on" per the config notes. I have enough to evaluate.

## Challenges

1. [RISK] [MATERIAL] [COST:MEDIUM]: **The proposal presents as net-new what is substantially already built.** `scripts/llm_client.py` already contains `_anthropic_call` (line 405), `_is_connection_refused` (line 168), `activate_fallback`, `is_fallback_active`, `get_fallback_model`, module-level `_FALLBACK_ACTIVE`/`_FALLBACK_WARNED` state, `_ANTHROPIC_API_URL`, and `_load_anthropic_key`. `tests/test_debate_fallback.py` (264 lines) already exists. `debate.py` has `_call_with_model_fallback`. The proposal's "What changes" section reads as if none of this existed. EVIDENCED by read_file_snippet and check_function_exists. The real unknown — and the actual scope of remaining work — is whether the multi-persona debate *commands* (`cmd_challenge`, `cmd_review`, `cmd_refine`) check `is_fallback_active()` and route persona calls through `_anthropic_call` when it returns true. The proposal should be rewritten as a *gap analysis* against existing code, not a greenfield design, or the PR will duplicate infrastructure.

2. [ASSUMPTION] [MATERIAL] [COST:TRIVIAL]: **"Cross-family disagreement" is named as the main tradeoff, but the current production config already has it disabled.** `config/debate-models.json` is labeled `arm-c-study-2026-04-17` with the note: *"all challengers Claude-only to isolate challenger-side model diversity."* Every persona in `persona_model_map` (architect, staff, security, pm, frame) is a Claude model; `refine_rotation` is Claude-only. EVIDENCED by read_config_value. So for the currently-deployed arm, "what you lose in fallback mode" is approximately *nothing* on the challenger side — only `judge_default: gpt-5.4` differs. The proposal overstates the delta. It should either (a) acknowledge the study context and scope fallback behavior to the non-arm-C baseline, or (b) note that under arm C the degradation is limited to the judge step.

3. [UNDER-ENGINEERED] [MATERIAL] [COST:SMALL]: **Tool-use path is a hard blocker the proposal doesn't mention.** `llm_tool_loop` at line 735 explicitly raises `LLMError("Tool-use loops require LiteLLM proxy — Anthropic fallback does not support tools")`. The config notes `frame_factual_model: claude-opus-4-7` runs "tools-on", and the `_run_claim_verifier` path in `debate.py` plus `debate_tools.py` use tool loops. So in fallback mode, the Frame persona's factual mode and claim verification silently lose functionality or hard-fail. The proposal's claim "Same interface... same structured output format... Downstream consumers don't need to know the difference" is false for any command that exercises tools. Either (a) port tool loops to Anthropic's native tool-use format, or (b) document and gracefully degrade these specific sub-features with a clear diff in the artifact frontmatter.

4. [RISK] [MATERIAL] [COST:TRIVIAL]: **Artifact indistinguishability is a downstream hazard, not a feature.** The proposal argues "output files have identical structure whether generated by 3 models or 1" so hooks don't need to know. But `debate_stats.py`, `debate_compare.py`, and the efficacy-study infrastructure exist precisely to measure cross-model effects. If fallback artifacts look identical, they will pollute `stats` aggregations and compare runs with single-model data masquerading as multi-model. A minimal fix: add a frontmatter field like `review_mode: fallback-single-model` and have `debate_common._build_frontmatter` emit it when `is_fallback_active()`. TRIVIAL to add; high value for data hygiene.

5. [ALTERNATIVE] [ADVISORY]: **Why urllib instead of the `anthropic` SDK?** The proposal hand-waves "Anthropic's SDK uses a different format" — but the codebase already imports the `openai` SDK lazily (`from openai import OpenAI`). Adding `from anthropic import Anthropic` inside `_anthropic_call` would get retries, streaming, tool-use, and error classes for free. The existing urllib implementation works but will diverge over time (e.g., prompt caching headers, tool-use request shape). Worth reconsidering before doubling down.

6. [UNDER-ENGINEERED] [ADVISORY]: **Fallback activation is sticky but never resets within a session.** `_FALLBACK_ACTIVE` becomes true on first connection-refused and stays true (there's a `_reset_fallback_state` but it's test-only). If a user starts the LiteLLM proxy mid-session, subsequent calls still route to Anthropic until process restart. Low severity, but worth a TTL or a "retry primary once per N minutes" policy since the proposal's narrative is about onboarding — and onboarding users are exactly the ones likely to start the proxy halfway through.

## Concessions

1. The user-facing problem is real and well-framed: LiteLLM is the heaviest setup step and gating `/review` on it is an onboarding cliff. The "single-model review > no review" argument is sound.
2. Keeping the persona system prompts applied under fallback (three lenses on one model) is the right design — it preserves the review *habit* even without model diversity.
3. Falling back on `ANTHROPIC_API_KEY` that Claude Code users already have is the correct credential choice.

## Verdict

**REVISE** — the feature is substantially already implemented in `llm_client.py` and `test_debate_fallback.py`; the proposal needs to be rescoped as a *wiring and gap-closure* task (persona-dispatch integration in `cmd_challenge`/`cmd_review`/`cmd_refine`, tool-loop degradation policy, artifact frontmatter tagging) rather than a greenfield design, and should correct its characterization of current cross-family diversity (arm C is Claude-only on challengers).

## Challenger B (security) — Challenges
## Challenges

1. **RISK [ADVISORY]**: The `_FALLBACK_ACTIVE` and `_FALLBACK_WARNED` globals are module-level state in `llm_client.py`. In concurrent execution contexts (debate.py uses `concurrent.futures` per line 51), a race condition exists where multiple threads could simultaneously detect LiteLLM failure and each attempt to set `_FALLBACK_ACTIVE = True` and print the warning. The `_FALLBACK_WARNED` dedup flag doesn't use a lock. This is low-severity (worst case: duplicate stderr lines), but worth noting. [COST: TRIVIAL]

2. **RISK [ADVISORY]**: The `.env` file parser in `_load_anthropic_key()` (lines 124-127) reads the file line-by-line and presumably does simple `KEY=VALUE` splitting. If the `.env` file contains quoted values, comments mid-line, or multiline values, the parser may silently fail to extract the key and fall through to an `LLMError`. This is a pre-existing issue, not introduced by this proposal, but the fallback path makes it newly load-bearing for users who store `ANTHROPIC_API_KEY` in `.env` rather than environment.

3. **ASSUMPTION [MATERIAL]**: The proposal states "ANTHROPIC_API_KEY is already available since the user is running Claude Code." This is not verified. Claude Code (Anthropic's CLI tool) uses its own credential store and does not necessarily export `ANTHROPIC_API_KEY` into the shell environment. A user running Claude Code via OAuth or API key stored in Claude's config directory would have no `ANTHROPIC_API_KEY` in their environment, causing the fallback to fail with a confusing error. The `check_code_presence` results confirm `ANTHROPIC_API_KEY` is only referenced in 2 places in scripts and 1 in docs — there's no detection or helpful error path for this case. [COST: SMALL — add detection + user-facing message pointing to where to get/set the key]

4. **RISK [ADVISORY]**: The `_anthropic_call` function uses `urllib.request` directly (confirmed at line 405). This means it bypasses the retry logic in `_call_with_retry` (which wraps the SDK path). If the Anthropic API returns a transient 529 (overloaded) or 500 during fallback mode, the call fails immediately with no retry. The primary path gets exponential backoff; the fallback path does not. This is an asymmetric reliability regression for the fallback case.

5. **ASSUMPTION [ADVISORY]**: The proposal claims "No degradation in artifact format — downstream consumers don't need to know the difference." However, if hooks or skills ever inspect the `model` field in artifact frontmatter (e.g., to enforce cross-family review requirements), they would silently accept single-model artifacts as equivalent to multi-model ones. The `_build_frontmatter` function in `debate_common.py` presumably records model metadata. This is a correctness assumption worth verifying but not a security issue.

6. **RISK [ADVISORY]**: Cost tracking (`_track_cost` in `debate_common.py`) uses model-name-based pricing lookup. In fallback mode, the model used is `_FALLBACK_MODEL` (e.g., `claude-sonnet-4-20250514`), which may or may not be in the `_COST_TABLE`. If it's missing, `_estimate_cost` returns 0.0 (per line 157 comment), so session cost reporting will silently undercount. Not a security issue, but a data quality issue.

7. **UNDER-ENGINEERED [MATERIAL]**: There is no rate-limit or cost guard on the fallback path. In fallback mode, `challenge` and `review-panel` run 3 lenses sequentially on the same Anthropic account. If a user runs `/review` in a loop (or a hook triggers it repeatedly), they're making direct Anthropic API calls with no circuit breaker, no cost cap, and no rate-limit handling (see finding #4 above). The LiteLLM proxy presumably provides rate limiting and cost controls; the fallback bypasses all of them. For a tool designed to run in automated hook pipelines, this is a meaningful gap. [COST: SMALL — add a per-session call counter and a configurable cap, defaulting to something reasonable like 20 calls]

8. **RISK [ADVISORY]**: The `_FALLBACK_MODEL` is set from `os.environ.get("ANTHROPIC_FALLBACK_MODEL", "claude-sonnet-4-20250514")` (line 149). This environment variable is not documented in the proposal, not mentioned in `docs/infrastructure.md` (per the single docs reference to `ANTHROPIC_API_KEY`), and not validated. A user who sets this to a non-Anthropic model name (e.g., `gpt-4o`) will get a confusing API error from the Anthropic Messages endpoint.

## Concessions

1. **The implementation already exists.** Tool calls confirm `_anthropic_call`, `is_fallback_active`, `activate_fallback`, `_FALLBACK_ACTIVE`, `_FALLBACK_WARNED`, and `_call_with_model_fallback` are all present in the codebase. This is a review of a completed implementation, not a proposal. The test file `tests/test_debate_fallback.py` (264 lines) exists and covers the fallback path. The proposal is describing work that's already done.

2. **The trust boundary is correctly drawn.** The fallback uses `urllib.request` directly to Anthropic's API rather than introducing a new dependency. The API key is loaded from the same source as the primary path. No new credential storage or transmission paths are introduced.

3. **The "no review vs. single-model review" framing is correct.** The structured lens system (PM/Security/Architecture) provides genuine value even with a single model. The proposal correctly identifies this as the dominant comparison for new users.

---

## Verdict

**REVISE** — The implementation is largely sound and already exists, but the MATERIAL assumption that `ANTHROPIC_API_KEY` is automatically available in Claude Code environments is unverified and likely false for a meaningful fraction of users. The fallback's value proposition collapses if the key isn't present and the error message isn't actionable. Fix: detect the missing-key case early (before the first LiteLLM failure), emit a specific message explaining how to set `ANTHROPIC_API_KEY` for fallback mode, and add a minimal rate-limit guard for the direct-API path. These are SMALL fixes that don't require architectural changes.

## Challenger D (frame-structural) — Challenges
## Challenges

1. **ASSUMPTION [MATERIAL] [COST:SMALL]:** The proposal assumes `ANTHROPIC_API_KEY` is always available in the environment because "the user is running Claude Code." This is not guaranteed — Claude Code can authenticate via OAuth/session tokens, browser-based auth, or enterprise SSO without exposing a raw API key to the shell environment. If the key isn't present, the fallback silently fails the same way LiteLLM does, and the user is no better off. The proposal should verify key availability as part of fallback detection, not assume it. SPECULATIVE claim: no data cited on what fraction of Claude Code users have `ANTHROPIC_API_KEY` set.

2. **ALTERNATIVE [MATERIAL] [COST:TRIVIAL]:** The proposal skips the most obvious candidate: **use `llm_client.py`'s existing `_anthropic_call` path** (already present per the exported functions manifest) rather than adding a new `urllib.request` raw HTTP path. The manifest shows `_anthropic_call`, `_sdk_call`, `activate_fallback`, `is_fallback_active`, and `get_fallback_model` already exist in `llm_client.py`. The fallback infrastructure is already partially built. The proposal's candidate set treats this as a greenfield addition when it may be a 20-line wire-up of existing machinery. This is a significant framing error — the proposal describes building what already exists.

3. **ASSUMPTION [MATERIAL] [COST:SMALL]:** The proposal states "The client already uses OpenAI-compatible API format; Anthropic's SDK uses a different format, so the fallback would use `urllib.request`." But `llm_client.py` already exports `_anthropic_call` and `_sdk_tool_call`, meaning the Anthropic SDK path is already implemented. The proposal's rationale for adding raw `urllib.request` HTTP calls is invalidated by the existing codebase. Adding a third HTTP path when two already exist is unnecessary complexity.

4. **ALTERNATIVE [ADVISORY] [COST:SMALL]:** A **degradation-level indicator in the artifact frontmatter** (e.g., `review_mode: single_model` vs `review_mode: multi_model`) is missing from the candidate set. The proposal explicitly says "No degradation in artifact format" as a feature, but this means downstream hooks and gates cannot distinguish a single-model review from a cross-model review. A gate that requires `/review` before shipping cannot enforce review quality. Adding one frontmatter field costs nothing and preserves the ability to enforce quality floors later.

5. **ASSUMPTION [ADVISORY]:** The proposal frames "every session where someone skips `/review` because LiteLLM isn't configured" as a significant failure mode, but provides no frequency data. SPECULATIVE: the severity claim ("single biggest onboarding cliff") is asserted without telemetry. `scripts/session_telemetry_query.py` and `scripts/lesson_events.py` exist in the codebase — if this were truly the top onboarding failure, there would be data. The absence of cited telemetry weakens the severity framing, though it doesn't invalidate the solution.

6. **OVER-ENGINEERED [ADVISORY]:** The proposal routes all three persona lenses (PM, Security, Architecture) sequentially through one model in fallback mode. A simpler fallback — one consolidated review prompt with all three lenses in a single call — would be cheaper (1 API call vs 3), faster, and equally useful for a degraded mode. The three-sequential-call approach preserves structural symmetry with the multi-model path but at 3x the cost and latency for no additional quality in single-model mode.

## Challenger E (frame-factual) — Challenges
## Challenges

1. **ASSUMPTION [MATERIAL] [COST:TRIVIAL]**: The proposal's entire premise — "If LiteLLM is unavailable, these features fail entirely" — is falsified by current code. `scripts/llm_client.py` lines 572-594 already implement the exact fallback described: on `_is_connection_refused` it loads `ANTHROPIC_API_KEY` via `_load_anthropic_key()`, calls `_anthropic_call()` against `https://api.anthropic.com/v1/messages`, sets `_FALLBACK_ACTIVE = True`, and prints to stderr the near-verbatim message "LiteLLM proxy unavailable — running single-model review.\nFor cross-model review, see docs/infrastructure.md". All of `activate_fallback`, `is_fallback_active`, `get_fallback_model`, `_anthropic_call`, `_load_anthropic_key`, and `_is_connection_refused` are already exported from `llm_client.py`. `tests/test_debate_fallback.py` (264 lines) covers `debate._call_with_model_fallback`. The proposal is re-specifying shipped code.

2. **OVER-ENGINEERED [MATERIAL] [COST:TRIVIAL]**: The proposal recommends writing `_anthropic_call` via `urllib.request` against the Messages API. That function already exists in `llm_client.py` (single definition, confirmed with `def _anthropic_call` match). Adding a second implementation would be pure duplication.

3. **ALTERNATIVE [MATERIAL] [COST:TRIVIAL]**: If the real gap the proposer cares about is *discoverability* (users not knowing fallback exists) or *a specific bug in the shipped fallback*, that's a materially different proposal — "document fallback in getting-started.md" or "fix concrete defect X in `_anthropic_call`." The current proposal doesn't name any defect in the shipped behavior, so there's nothing to build. The author appears to have not read `llm_client.py` before writing.

4. **RISK [ADVISORY]**: One area the shipped fallback may not cover is multi-persona rotation inside `debate.py`'s `cmd_challenge`/`cmd_review`/`cmd_refine` — the proposal's claim that "in fallback mode, route all persona/model requests to the single available model" is worth verifying against `_call_with_model_fallback` and `refine_rotation`. ## Concessions

1. The value framing (single-model-through-3-lenses >> no-review) is sound and aligns with what the shipped fallback actually delivers.
2. The stderr message format and the "no artifact format change" principle match the existing implementation — suggesting the design instinct is correct, just uninformed about prior work.
3. Keeping `config/debate-models.json` unchanged is correct; the shipped fallback also bypasses routing.

## Verdict

**REJECT.** The feature is already implemented in `scripts/llm_client.py` (connection-refused detection → Anthropic direct via `ANTHROPIC_API_KEY`, with the exact stderr message the proposal specifies) and covered by `tests/test_debate_fallback.py`. The proposal needs to either (a) identify a concrete defect in the current fallback, (b) target the discoverability/docs gap, or (c) be withdrawn. As written, it would duplicate shipped code.
