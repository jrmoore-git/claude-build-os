```
# Proposal: LiteLLM Graceful Degradation — Single-Model Fallback

## Problem

BuildOS's most differentiated features — `/challenge` and `/review` — require a running LiteLLM proxy (localhost:4000) with three API providers (Anthropic, OpenAI, Google) configured. If LiteLLM is unavailable, these features fail entirely. This is the single biggest onboarding cliff in BuildOS.

Every session where someone skips `/review` because LiteLLM isn't configured is a session where they get B-tier output from an A-tier framework. The review pipeline creates the most value over raw Claude Code, but it's gated behind the heaviest infrastructure requirement.

## Proposed Solution

When `debate.py` detects LiteLLM is unavailable (connection refused on localhost:4000), fall back to single-model review using the Anthropic API directly (which is already available since the user is running Claude Code).

### Behavior

1. **Detection:** On first LLM call failure (connection refused / timeout to LiteLLM proxy), detect and switch to fallback mode.
2. **Fallback routing:** Use the Anthropic API directly (via `ANTHROPIC_API_KEY` from environment, which Claude Code already has configured) with the session's own Claude model.
3. **Same interface:** Run the same three lenses (PM, Security, Architecture) sequentially with the same structured output format. Write the same artifact files (`tasks/<topic>-challenge.md`, `tasks/<topic>-review.md`).
4. **Transparency:** Print one line to stderr: "LiteLLM unavailable — running single-model review. For cross-model review, see docs/infrastructure.md"
5. **No degradation in artifact format:** The output files have identical structure whether generated by 3 models or 1. Downstream consumers (hooks, skills) don't need to know the difference.

### What changes

- `scripts/llm_client.py`: Add fallback path that calls Anthropic API directly when LiteLLM connection fails. The client already uses OpenAI-compatible API format; Anthropic's SDK uses a different format, so the fallback would use `urllib.request` with Anthropic's Messages API.
- `scripts/debate.py`: In multi-model commands (`challenge`, `review-panel`, `judge`, `refine`), when fallback mode is active, route all persona/model requests to the single available model. The persona system prompts still apply — each lens still gets its framing (PM, Security, Architecture) — but they all run on the same model.
- `config/debate-models.json`: No changes. The fallback bypasses model routing entirely.

### What does NOT change

- Hook behavior (all hooks work the same)
- Artifact format (same YAML frontmatter, same finding structure)
- Skill interfaces (skills read artifacts, not model metadata)
- Full multi-model path (when LiteLLM is available, everything works as before)
- The `refine` command's model rotation (falls back to single-model iterative improvement)

### Tradeoffs

**What you lose in fallback mode:**
- Cross-family disagreement (Claude, GPT, Gemini catch different things)
- Independent evaluation (single model has correlated blind spots)
- Model diversity in refinement rounds

**What you keep:**
- Structured review through PM/Security/Architecture lenses
- Artifact generation for hooks and gates
- The review habit — users who start with single-model review will upgrade to multi-model when they see value

**Why this is acceptable:** A single-model review through 3 structured lenses is dramatically better than no review. The difference between "no review" and "single-model review" is much larger than the difference between "single-model review" and "cross-model review."

## Expected Impact

- **New users:** Can run `/review` and `/challenge` on first session, zero additional setup
- **Existing users:** No change when LiteLLM is running
- **Onboarding path:** Clone → setup → `/think` → `/plan` → build → `/review` works end-to-end with just Claude Code installed
- **Upgrade path:** User sees "LiteLLM unavailable" message, eventually sets up multi-model, gets better reviews — natural discovery
```

## DECISION DRIVERS

**Outcome:** REJECTED by frame lens (ground-truth-v2 confirms). Feature was already shipped before the proposal was written.

### Decision drivers

**DD-1: Feature already shipped via commit `37440d9` on 2026-04-12.**  
Commit title: "LiteLLM graceful degradation: single-model fallback via ANTHROPIC_API_KEY." Proposal dates to 2026-04-16 (4 days later). This is the single most important driver. *Evidence: `git show 37440d9` + proposal frontmatter date.*

**DD-2: Proposal's ANTHROPIC_API_KEY availability assumption is wrong for all paths.**  
Proposal assumes ANTHROPIC_API_KEY is available. Not all tool-enabled code paths access this; some route through LiteLLM exclusively. *Evidence: grep of `_sdk_tool_call` and `_anthropic_call` show mixed patterns.*

**DD-3: Transport-layer fallback conflates model-family independence with API-independence.**  
A fallback that silently routes from gemini-3.1-pro to anthropic breaks model-family-independence invariant of cross-family debate. *Evidence: D21 establishes cross-family requirement for judge; fallback could violate it.*

**DD-4: Fallback trigger condition (timeout vs rate-limit vs API-error) was underspecified.**  
Proposal doesn't distinguish which failure modes trigger fallback vs retry. *Evidence: proposal text absence of precise trigger spec.*

**DD-5: Judge independence contract would be broken by transparent fallback.**  
If judge (gpt-5.4) fails over to another model silently, the "judge ≠ any challenger" contract can be violated without warning. *Evidence: D21 + current judge-overlap warning in debate.py.*

## Finding 16

**RISK [ADVISORY]**: The `_anthropic_call` function uses `urllib.request` directly (confirmed at line 405). This means it bypasses the retry logic in `_call_with_retry` (which wraps the SDK path). If the Anthropic API returns a transient 529 (overloaded) or 500 during fallback mode, the call fails immediately with no retry. The primary path gets exponential backoff; the fallback path does not. This is an asymmetric reliability regression for the fallback case.

## Finding 20

## Finding 41

[RISK] [MATERIAL] [COST:TRIVIAL]: **Artifact indistinguishability is a downstream hazard, not a feature.** The proposal argues "output files have identical structure whether generated by 3 models or 1" so hooks don't need to know. But `debate_stats.py`, `debate_compare.py`, and the efficacy-study infrastructure exist precisely to measure cross-model effects. If fallback artifacts look identical, they will pollute `stats` aggregations and compare runs with single-model data masquerading as multi-model. A minimal fix: add a frontmatter field like `review_mode: fallback-single-model` and have `debate_common._build_frontmatter` emit it when `is_fallback_active()`. TRIVIAL to add; high value for data hygiene.

## Finding 42

**ASSUMPTION [MATERIAL] [COST:SMALL]:** The proposal assumes `ANTHROPIC_API_KEY` is always available in the environment because "the user is running Claude Code." This is not guaranteed — Claude Code can authenticate via OAuth/session tokens, browser-based auth, or enterprise SSO without exposing a raw API key to the shell environment. If the key isn't present, the fallback silently fails the same way LiteLLM does, and the user is no better off. The proposal should verify key availability as part of fallback detection, not assume it. SPECULATIVE claim: no data cited on what fraction of Claude Code users have `ANTHROPIC_API_KEY` set.

## Finding 43

**ASSUMPTION [MATERIAL] [COST:TRIVIAL]**: The proposal's entire premise — "If LiteLLM is unavailable, these features fail entirely" — is falsified by current code. `scripts/llm_client.py` lines 572-594 already implement the exact fallback described: on `_is_connection_refused` it loads `ANTHROPIC_API_KEY` via `_load_anthropic_key()`, calls `_anthropic_call()` against `https://api.anthropic.com/v1/messages`, sets `_FALLBACK_ACTIVE = True`, and prints to stderr the near-verbatim message "LiteLLM proxy unavailable — running single-model review.\nFor cross-model review, see docs/infrastructure.md". All of `activate_fallback`, `is_fallback_active`, `get_fallback_model`, `_anthropic_call`, `_load_anthropic_key`, and `_is_connection_refused` are already exported from `llm_client.py`. `tests/test_debate_fallback.py` (264 lines) covers `debate._call_with_model_fallback`. The proposal is re-specifying shipped code.

## Finding 44

**ASSUMPTION [MATERIAL] [COST:SMALL]:** The proposal treats "same artifact format = no downstream impact" as self-evident, but hooks like `hook-post-build-review.py` and `hook-spec-status-check.py` may read metadata fields (model provenance, reviewer count, confidence signals) that would differ or be absent in single-model output. The claim "downstream consumers don't need to know the difference" is asserted, not verified. If any hook gates on multi-model consensus signals, the fallback artifacts could silently pass gates they shouldn't.

## Finding 45

**RISK** [MATERIAL] [COST:SMALL]: The real remaining gap is that `debate.py` is not fallback-*aware*. A grep of `scripts/` shows `is_fallback_active` has exactly 1 occurrence — its definition in `llm_client.py` — so `debate.py` never inspects it. In fallback mode, `/challenge` with `architect,security,pm` still issues three calls with three distinct model names (`claude-opus-4-7`, `gpt-5.4`, `gemini-3.1-pro` per `persona_model_map`); the transport layer transparently routes all three to the *same* Anthropic model, so the artifact will claim cross-model diversity that doesn't exist. The judge and refine commands may double-count correlated findings as "corroboration" across what they think are independent models. Fix: after first call detects fallback, stamp artifact frontmatter with `review_mode: single-model-fallback` and skip the consolidation step that assumes model independence. This is the missing 20% the proposal should focus on.

## Finding 46

[RISK] [ADVISORY]: Credential handling is acceptable but increases secret exposure surface slightly because `llm_client.py` reads `ANTHROPIC_API_KEY` directly from both environment and `.env`. That is not inherently flawed, but adding direct-provider calls means one more code path that can accidentally log headers, request bodies, or error payloads if future debugging is added. I did not see evidence in the inspected snippets of secrets being printed today, so this is advisory: keep explicit tests/assertions that auth headers and raw request objects never reach stderr/logs.

## Finding 47

**OVER-ENGINEERED [ADVISORY]:** The proposal mentions `urllib.request` for the Anthropic fallback path specifically to avoid adding the `anthropic` SDK as a dependency. This is a reasonable constraint, but the existing `_anthropic_call` already uses `urllib.request` (confirmed at line 384-430). The proposal is re-describing an already-made design decision, not proposing one.

## Finding 48

[ASSUMPTION] [MATERIAL] [COST:SMALL]: The proposal claims `/challenge` and `/review` “fail entirely” if LiteLLM is unavailable, but current behavior contradicts that. `debate.py` loads credentials through `debate_common._load_credentials()` (`scripts/debate.py:3042-3045`), and that helper explicitly allows Anthropic-only operation by activating fallback (`scripts/debate_common.py:81-91`). Since `llm_client._dispatch()` then routes calls to `_anthropic_call()` in fallback mode (`scripts/llm_client.py:541-560`), the stated onboarding cliff is overstated. If there is still a UX problem, it is narrower than the proposal says and should be reframed around commands/features that still degrade poorly.
