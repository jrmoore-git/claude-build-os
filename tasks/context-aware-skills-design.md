# Design: Context-Aware Skill Invocation

Generated by session observation on 2026-04-13
Branch: main
Status: DRAFT
Mode: Builder

## Problem Statement

Skills assume every invocation is a cold start. `/think discover` runs the same 5-question intake sequence whether the user just walked in or has been working the problem for an hour. When a problem emerges organically from conversation (as it did with model routing — 20 minutes of discussion before `/think` was invoked), the intake questions are redundant friction.

This was observed directly: `/think discover` asked "what's the coolest version of this?" and "where's the main value?" for a problem where the problem statement, evidence (model comparison data), and candidate approach (config-driven routing) were all already established in the conversation.

## Root Cause

Skills are stateless procedures. They don't read the conversation they're in — they follow a fixed sequence regardless of available context. The skill SKILL.md files are prompts, not programs. They can't introspect on what the caller already knows.

## What Makes This Cool

Skills that feel like smart colleagues instead of forms. You say `/think` and it says "I've been following along — here's what I have so far, and here are the 2 open questions I can't answer from context." Instead of starting over.

## Approaches Considered

### Approach A: Pre-scan conversation context in skill preamble

Add a "context assessment" phase to the skill preamble (before the first question). The skill reads the last N messages / recent tool results and classifies how much of its intake is already answered:

- Problem statement present? → skip problem-gathering questions
- Evidence/examples present? → skip diagnostic questions  
- Approach discussed? → skip alternatives generation, go straight to synthesis
- Open questions identifiable? → ask only those

Implementation: add a "Phase 0: Context Assessment" to each skill's SKILL.md that instructs Claude to scan conversation context before proceeding.

**Effort:** S per skill — add ~10 lines of preamble instruction
**Risk:** Low — worst case it still asks questions (false negative on context detection)

### Approach B: Structured context handoff

When invoking a skill mid-conversation, the caller writes a context summary to a temp file that the skill reads. Like passing arguments but richer.

**Effort:** M — needs caller-side changes + skill-side reading
**Risk:** Med — adds coupling between caller and skill, context summaries can be lossy

### Approach C: Conversation-aware skill router

A meta-layer that intercepts skill invocations, scans conversation context, and rewrites the skill's procedure to skip phases that are already satisfied.

**Effort:** L — new infrastructure layer
**Risk:** High — brittle, hard to debug, over-engineered

## Recommended Approach

**Approach A** — it's the simplest and works within the existing skill architecture. Each skill gets a Phase 0 that says:

```
Before starting the intake sequence, assess what context already exists 
in this conversation:

1. Is the problem statement clear from prior discussion? 
   If yes: state it in one line, skip problem-gathering questions.
2. Is there evidence or examples already discussed?
   If yes: reference them, skip diagnostic questions.
3. Has an approach been discussed or proposed?
   If yes: reference it, skip alternatives unless the user wants more options.
4. What questions remain unanswered?
   Ask ONLY those.

If all phases are satisfied by conversation context, go directly to 
document synthesis. Present the doc and ask: "Anything I'm missing 
or got wrong?"
```

This is a prompt-level fix, not an architectural one. It works because Claude can read its own conversation context — it just needs to be told to check before asking.

## Open Questions

1. Should this be a shared preamble (one file included by all skills) or per-skill?
2. Does this interact with the governance context pull (Phase 1.5)? Probably independent.
3. Should the context assessment be explicit ("I found X, Y, Z in our conversation") or silent (just skip questions)?

## Success Criteria

- `/think discover` invoked mid-conversation skips intake questions that conversation already answers
- Time from `/think` invocation to design doc: < 2 minutes when context is rich (vs 10-15 min cold start)
- No regression on cold-start quality — when context is absent, full intake still runs
- User never has to say "you already know this" during a skill
