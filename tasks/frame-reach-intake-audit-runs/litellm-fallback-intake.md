# litellm-fallback-intake — Challenger Reviews

## Challenger A (frame-structural) — Challenges
## Challenges

1. **ASSUMPTION [MATERIAL] [COST:SMALL]:** The proposal assumes the Anthropic API key is always available in the environment because "the user is running Claude Code." This is not guaranteed — Claude Code can run via OAuth/session tokens, enterprise SSO, or API-key-less configurations. The fallback path silently fails if `ANTHROPIC_API_KEY` is absent, which is exactly the same failure mode as LiteLLM being down. The proposal needs a detection step that confirms the fallback credential is actually present before advertising this as a zero-setup path. SPECULATIVE claim: "which is already available since the user is running Claude Code."

2. **ALTERNATIVE [MATERIAL] [COST:TRIVIAL]:** The candidate set is binary: full multi-model LiteLLM OR single-model Anthropic fallback. A missing third option is **single-model via LiteLLM itself** — if LiteLLM is running but only one provider is configured (e.g., only Anthropic), the existing `_call_with_model_fallback` path in `debate_common.py` may already handle this. The proposal doesn't evaluate whether the real onboarding cliff is "LiteLLM not running at all" vs. "LiteLLM running but missing 2 of 3 providers." These require different fixes, and the partial-LiteLLM case may be far more common among users who got partway through setup.

3. **ASSUMPTION [MATERIAL] [COST:MEDIUM]:** The proposal claims "artifact format is identical whether generated by 3 models or 1" and that "downstream consumers don't need to know the difference." But `hook-post-build-review.py` and `scripts/artifact_check.py` both parse frontmatter and check artifact freshness. If the fallback artifacts omit model provenance metadata (which multi-model artifacts presumably include), hooks that gate on review quality or model diversity will silently accept lower-quality artifacts as equivalent. The proposal explicitly says `config/debate-models.json` is unchanged — meaning there's no flag in the artifact indicating fallback mode was used. This is a correctness risk for any future gate that distinguishes review quality tiers.

4. **ALTERNATIVE [ADVISORY]:** A simpler compositional option not considered: **document the single-model workaround as a manual invocation pattern** (e.g., `debate.py challenge --single-model`) rather than auto-detecting and silently switching. Auto-detection with a single stderr line buries the degradation signal. An explicit flag makes the tradeoff visible at invocation time, avoids the detection complexity, and doesn't require changes to `llm_client.py`'s retry/fallback logic. The `is_fallback_active` / `activate_fallback` functions already exist in `llm_client.py` — this may be a one-flag addition to `debate.py` rather than a new detection path.

5. **OVER-ENGINEERED [ADVISORY]:** The proposal routes through `urllib.request` with Anthropic's Messages API directly because "Anthropic's SDK uses a different format." But `llm_client.py` already has `_anthropic_call` and `_sdk_call` functions that handle this. The proposal appears to be re-implementing what already exists. This suggests the author may not have fully read `llm_client.py`'s existing fallback infrastructure (`activate_fallback`, `get_fallback_model`, `_call_with_model_fallback`) before proposing new code. The real change may be much smaller than described.

6. **RISK [ADVISORY]:** The "same three lenses sequentially on one model" design has a known failure mode: a single model asked to play PM, then Security, then Architecture in sequence will exhibit anchoring — each subsequent persona is influenced by what the prior persona said. Multi-model review's value comes partly from independence, not just persona framing. The proposal acknowledges correlated blind spots but doesn't acknowledge the anchoring problem, which means the fallback quality claim ("dramatically better than no review") may be overstated for adversarial/security findings specifically.

---

## Concessions

1. **The problem is real and correctly diagnosed.** LiteLLM as a hard dependency for the most valuable feature is a genuine onboarding cliff, not an inflated severity claim. The proposal correctly identifies that the gap between "no review" and "single-model review" is larger than "single-model" vs. "multi-model."

2. **The existing infrastructure is well-understood.** The proposal correctly identifies which files change (`llm_client.py`, `debate.py`) and which don't (hooks, skills, artifact format), and the scope is appropriately narrow.

3. **The upgrade path is sound.** Surfacing "LiteLLM unavailable" at runtime with a pointer to docs is the right discovery mechanism — better than silent failure or a setup wizard.

---

## Verdict

**REVISE** — The proposal's frame is mostly sound but has two material gaps: (1) it doesn't verify the fallback credential is actually available before claiming zero-setup, and (2) it doesn't evaluate whether the existing `activate_fallback` / `_anthropic_call` infrastructure in `llm_client.py` already covers most of this, making the `urllib.request` re-implementation potentially unnecessary. Both gaps should be resolved before implementation begins, as they could reduce the actual code change from MEDIUM to TRIVIAL.

## Challenger B (frame-factual) — Challenges
## Challenges
1. [ASSUMPTION] [MATERIAL] [COST:TRIVIAL]: The proposal is stale on its core premise: `scripts/llm_client.py` already implements direct Anthropic fallback when LiteLLM is unavailable. `_dispatch()` detects connection-refused errors, loads `ANTHROPIC_API_KEY`, flips `_FALLBACK_ACTIVE`, emits the exact warning, and routes to `_anthropic_call(...)` using `_FALLBACK_MODEL` (`scripts/llm_client.py:536-594`, `143-149`). The repo also already exposes `is_fallback_active`, `get_fallback_model`, and `activate_fallback`, so “add fallback path” is not net-new work.

2. [ASSUMPTION] [MATERIAL] [COST:SMALL]: The proposal claims debate.py must be updated so multi-model commands route all personas to one model in fallback mode, but the current implementation does not do that uniformly, and the proposal overstates command coverage. `review` still resolves persona-specific models from config and invokes them in parallel (`scripts/debate.py:3161-3244`), relying on the client fallback underneath rather than explicitly collapsing the panel. Also, the proposal names `review-panel`, which does not exist as a subcommand; the actual command is `review` (`scripts/debate.py:5-14`, `check_code_presence("review-panel")` only finds the proposal text, not an implemented command). So the proposal is partly describing work as if absent, and partly naming interfaces that aren’t real.

3. [OVER-ENGINEERED] [ADVISORY]: The proposal recommends adding custom `urllib.request` Anthropic handling because “the client already uses OpenAI-compatible API format,” but `llm_client.py` already has a dedicated `_anthropic_call` path and tests enforcing that all dispatch paths, including Anthropic, flow through shared sanitization (`tests/test_llm_client.py:282-299`). Re-proposing a separate fallback transport duplicates shipped infrastructure rather than extending the existing client behavior.

## Concessions
- The motivation is directionally valid: LiteLLM unavailability is a meaningful failure mode, and single-model degradation is a reasonable UX goal.
- The desired transparency message matches what the client already prints on fallback (`scripts/llm_client.py:551-556`, `585-590`).
- Keeping artifact formats stable is a sound requirement; nothing in the current fallback path suggests downstream artifact consumers should need format changes.

## Verdict
REVISE — the proposal’s central claim is already implemented in `llm_client.py`, so it needs to be reframed around the remaining gap in `debate.py` behavior rather than proposing fallback from scratch.
