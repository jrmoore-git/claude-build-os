# Design: Intelligent Model Routing for the Debate Engine

Generated by /think discover on 2026-04-13
Branch: main
Status: DRAFT
Mode: Builder

## Problem Statement

The debate engine routes personas to models via config but has no task-type-based routing. When a skill or Claude needs a single model's opinion, the model choice is ad-hoc — whoever's calling picks whatever comes to mind. We observed in this session that different models have measurably different strengths (GPT catches implementation blockers, Gemini finds edge cases, Claude does strategic coherence, Perplexity has the web). No one makes that choice deliberately.

## What Makes This Cool

`review --model auto` — you don't say which model, the tool figures it out. The model becomes invisible, just like the subcommand became invisible in the review unification. And when the answer requires knowledge outside the codebase, it knows to call Perplexity without being told.

## Premises

1. Different models have different strengths that are stable enough to route on (evidence: GPT caught argparse blocker that others missed, Gemini produced 13 vs 5 items on review)
2. Task type can be classified cheaply from the input document + prompt (frontmatter, keywords, file path patterns)
3. The routing should be config-driven so it survives model upgrades (re-benchmark when GPT-6/Gemini-4/Claude-5 ships — already noted in debate-models.json)
4. Perplexity is a model in the routing table, not a separate system — it gets routed to when the task needs external knowledge

## Approaches Considered

### Approach A: Config-driven task_routing map + `auto` flag

Add a `task_routing` section to `debate-models.json`:

```json
"task_routing": {
  "implementation_review": "gpt-5.4",
  "completeness_review": "gemini-3.1-pro",
  "strategic_review": "claude-opus-4-6",
  "web_research": "perplexity/sonar-pro",
  "classification": "claude-haiku-4-5",
  "default": "gpt-5.4"
}
```

Add `--model auto` (or make it the default when `--model` is passed without a value) that does a lightweight heuristic classification of the input:
- Frontmatter `type:` field → direct lookup
- File path patterns (`*-plan.md`, `*-diff.md`, `*-proposal.md`) → task type inference
- Prompt keyword scan → fallback classification
- If ambiguous → use `default`

No LLM call for classification — pure heuristic. Fast, deterministic, zero cost.

**Effort:** S — config change + ~50 lines of routing logic
**Risk:** Low — heuristic misclassification just means suboptimal model, not failure

### Approach B: Haiku classifier + routing table

Same config map, but classification is done by a Haiku call instead of heuristics. Send the first 500 chars of input + the prompt to Haiku, ask it to classify into one of the task_routing categories.

**Effort:** M — needs Haiku call, async handling, fallback on timeout
**Risk:** Med — adds latency (~1s), cost (~$0.001), and a failure mode (Haiku down)

### Approach C: Hybrid — heuristic first, Haiku fallback

Try heuristic classification first. If confidence is high (frontmatter match or clear file pattern), use it. If ambiguous, fall back to Haiku classifier.

**Effort:** M — both paths needed
**Risk:** Low — heuristic handles 80% of cases, Haiku only fires on ambiguous inputs

## Recommended Approach

**Approach A** (config-driven heuristic). The heuristic handles the common cases well enough — the debate system's inputs are structured (frontmatter, file naming conventions, predictable prompt patterns). The marginal accuracy gain from Haiku doesn't justify the latency and complexity. If heuristic accuracy proves insufficient, upgrading to Approach C is additive.

For panel mode (`--models auto`), the routing table maps task types to model *combinations*:

```json
"panel_routing": {
  "code_review": ["claude-opus-4-6", "gpt-5.4", "gemini-3.1-pro"],
  "document_review": ["gemini-3.1-pro", "gpt-5.4"],
  "investigation": ["claude-opus-4-6", "gemini-3.1-pro", "gpt-5.4"]
}
```

## Open Questions

1. Should `auto` be the default when no model is specified, or require explicit `--model auto`?
2. Should Perplexity routing be automatic (detect "needs web knowledge" from prompt) or require a keyword/flag?
3. Do we need per-skill routing overrides, or is the global config enough?

## Success Criteria

- `debate.py review --model auto --prompt "Review this for implementability" --input plan.md` routes to GPT
- `debate.py review --model auto --prompt "Is this strategy sound?" --input proposal.md` routes to Claude
- `debate.py review --models auto --prompt "Review this" --input diff.md` routes to the 3-model code review panel
- Routing decision logged to stderr so it's visible but not in output
- Config change only (no code change) when model preferences shift
