Mistakes, surprises, and patterns worth remembering. Each entry is a lesson learned from real work on this project.

**Convention:** Titles are assertions, not topics. "Long sessions silently eat their own corrections — compact at 50-70% context" — not "Session quality." The title IS the takeaway.

**Target:** Keep this file under 30 active entries. When it grows beyond that, promote recurring lessons to `.claude/rules/` files and archive one-offs.

| L41 | Shipping a hook file is not shipping enforcement — `.git/hooks/pre-commit` symlink is per-clone state and not in the repo | Bundle-2 QA discovered the anti-slop hook was not firing on real `git commit`. | L44 | N=1 is not data — LLM tuning decisions must ground in paired output-quality comparisons across n≥3 (ideally 5+) representative inputs. Quality first, speed/cost second. | Frame-lens development (2026-04-17) almost shipped a cross-family model change (frame-factual: sonnet → gpt-5.4) based on a single gbrain run showing 3x speedup and apparent finding diversity. One data point is consistent with any story — model difference, prompt difference, proposal difficulty, randomness. Corrected by running the same pair across n=5 historical proposals: cross-family was BETTER on 4/5 (more architectural findings, higher precision per finding, better synthesis of staleness implications) and TIED on 1/5. The 3x speedup turned out to be real but secondary — leading with speed was the wrong frame. Same pattern earlier in the session: "tools bias frame critique" was formed on n=1 (gbrain tools-on vs tools-off) and only became credible after n=4 paired validation. | Before shipping any LLM tuning decision (model swap, prompt edit, threshold change), the evidence bar is paired comparisons on n≥3 (ideally 5+) representative inputs, scored on **output quality** — not on speed, cost, or token count. Speed/cost are secondary considerations once quality is established. Quality dimensions to compare: precision per finding (signal-to-noise), recall on known-defect proposals, verdict alignment with ground truth, and whether the change catches a *kind* of issue the prior config systematically missed. "It was faster" or "it produced more findings" are not quality measures. The asymmetry: 5 parallel calls cost ~30-60s and pennies; shipping a bad tuning decision (or a good one for the wrong reason) costs hours of debugging. Applies to: model selection, persona prompt changes, tool-access defaults, temperature adjustments, threshold tuning. |
| L47 | Iterating an LLM prompt without a baseline benchmark produces a prompt-engineering treadmill — qualitative "looks cleaner" is not evidence of improvement | Refine-stage Frame Check directive went v1→v5 over 5 iterations (2026-04-18). I called v3, v4, and v5 all "calibrated" based on pattern-matching on individual outputs. When a proper benchmark was finally built (same 5 proposals, same metrics, ground-truth from judge-stage findings), it revealed v2 had 0/6 detection rate (silent failure I hadn't measured), and v3/v4 had channel violations I'd dismissed as "minor edge cases." The user caught the methodology gap by asking "did you benchmark this against the 5 reviews we had done before?" — I had not. A proper benchmark would have caught calibration errors on iteration 2, not iteration 5. This is the same failure mode L44 warned about (n=1 for tuning decisions), applied to prompt iteration rather than model selection. | When iterating a prompt: establish test set + metrics + ground truth BEFORE v2. Re-run the benchmark each iteration and compare explicitly — detection rate, false-positive rate, channel-violation count, whatever the directive promises. Qualitative "looks cleaner" is pattern-matching; it can move problems around without fixing them. Benchmark discipline must precede iteration, not follow it. Applies to any LLM tuning (model selection, prompt edits, threshold changes). If a benchmark isn't feasible (cost, setup time), limit iteration to 2 rounds — don't chase "just one more tweak" without evidence. |
| L46 | Frame-critique directives added to LLM prompts MUST include an explicit novelty gate — else the model reframes existing findings as "frame" findings and inflates MATERIAL counts without adding signal | Judge-stage FRAME CRITIQUE shipped 2026-04-18 as an addition to `JUDGE_SYSTEM_PROMPT`. Both pointed at the SAME required fix (gate the output). The frame finding was real, not fabricated, but redundant — the concern was covered by an existing challenger finding. Left uncorrected, this pattern inflates frame-value perception, and users learn to discount frame findings. | Before shipping any frame-critique directive on an LLM prompt: include a NOVELTY TEST that requires the model to (1) state the concrete FIX the frame finding implies, (2) scan existing findings for any recommending the same fix, (3) suppress the new finding if the fix is covered. Output format must separate "additive" from "covered" findings, with the covered ones tagged to the specific challenge that covers them. **Test for the gate:** run on a negative-control proposal (simple, deliberately-clean) WITH its generated 3-persona challenge. If frame MATERIAL findings appear that reframe existing challenge concerns, the novelty gate is too loose. Evidence: `tasks/judge-frame-directive-validation/` — first 11 runs without novelty gate showed the redundancy pattern; 3 novelty-gate runs show explicit "covered by Challenge [N]" mapping for reframes. **Generalization:** the same pattern will apply to refine-stage, review-lens, and any other LLM prompt where the model is asked to critique input it already processed through another lens.
