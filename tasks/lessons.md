# Lessons

Mistakes, surprises, and patterns worth remembering. Each entry is a lesson learned from real work on this project.

**Convention:** Titles are assertions, not topics. "Long sessions silently eat their own corrections — compact at 50-70% context" — not "Session quality." The title IS the takeaway.

**Target:** Keep this file under 30 active entries. When it grows beyond that, promote recurring lessons to `.claude/rules/` files and archive one-offs.

---

| # | Lesson | Source | Rule |
|---|---|---|---|
| L41 | Shipping a hook file is not shipping enforcement — `.git/hooks/pre-commit` symlink is per-clone state and not in the repo | Bundle-2 QA discovered the anti-slop hook was not firing on real `git commit`. All 10 session commits passed silently through the "enforced" hook because `.git/hooks/pre-commit` was never symlinked to `hooks/pre-commit-banned-terms.sh`. Hook file shipped and worked when run manually (`bash hooks/...`), but the git chain wasn't wired. Install step is documented in the hook's header comment, but relies on each developer running `ln -sf` once. | When promoting a rule to a pre-commit hook, the promotion is incomplete until the symlink exists. For a per-clone installation: either (a) add the symlink step to `setup.sh`, (b) add a session-start check that warns when an enforcement hook's symlink is missing, or (c) use git's `core.hooksPath` config (set once, applies repo-wide, tracked in repo config). Generalize: any "enforcement" that lives in per-developer filesystem state should have a matching installer. |
| L28 | Cross-model panels converge on wrong answers when operational evidence is missing from input | 3 independent models all said "sim-compiler is commodity, don't build" — unanimous and wrong. Root cause: proposal lacked eval_intake.py's track record (17 rounds, 3.3→4.5 improvement). Models said "DeepEval does this" without anyone checking whether DeepEval actually meets the specific requirements (it doesn't — no structured persona hidden state, hardcoded simulator prompts, no agent procedure testing). Absence of bugs was treated as absence of need (survivorship bias). Dramatic quantitative findings (18/22 missing Safety Rules) crowded out qualitative evidence (simulation proved its value). | Challenge inputs must include operational evidence when it exists. "Commodity" claims about external tools must be verified against specific requirements before influencing judgment, not just category-matched. Unanimous convergence from models with the same incomplete context is correlated error, not strong consensus. |
| L30 | V2 pipeline quality gap is structural (persona format + mid-loop interventions), not architectural — and the fix isn't more automation | Built full V2 pipeline (1,520 lines, 151 tests), scored 3.11/5 vs eval_intake's 4.73. Added turn_hooks (spike), got to 3.70. Three root causes resist generalization: thin persona cards, skill-specific register rules, hooks bolted on rather than co-evolved. Cross-model panel (3 models) converged: eval_intake's quality came from 17 iterations of watching-and-fixing, not from any architectural decision. Front-loading knowledge via questionnaire won't work because humans are better critics than oracles. | Simulation quality comes from iterative refinement on concrete failures, not from upfront configuration or automation. When building evaluation frameworks, design for critique loops (run → review transcript → annotate failures → adjust → rerun) not for zero-setup automation. |
| L31 | Rubric dimensions must measure product outcomes not interaction style | Spike optimization focused on register_match and flow — style dimensions. The actual value question is: did the skill identify the real problem, reach a useful conclusion, help the user make a better decision? Style dims are noise that distracts from substance. | When building rubrics or evaluation criteria for skills, frame dimensions around: problem identification, conclusion quality, decision support, time efficiency. Drop tone/register/flow dimensions unless the skill is specifically about communication. |
| L32 | Runtime directive injection degrades simulation quality — eval_intake worked by changing skill prompts, not runtime context | D22 critique loop spike: hand-crafted directives injected every turn scored 1.78 avg vs 2.50 baseline (degradation, not improvement). 3-model pre-mortem converged on "wrong mechanism" — eval_intake's 17 iterations changed SKILL.md text directly, not runtime executor context. Injecting directives adds cognitive load to the executor without changing its core instructions. | Simulation quality improvements must modify the skill prompt itself (the artifact being tested), not the simulation runtime. If building future critique loops, the output of each iteration should be a SKILL.md diff, not a runtime directive. |
| L33 | debate.py --enable-tools produces false verification claims with high confidence — deterministic checks beat probabilistic review for verifiable facts | Challenge session 18: Claude Opus ran 54 tool calls (125s, hit max turns) and falsely claimed BuildOS hooks don't exist (searched scripts/ instead of hooks/). GPT-5.4 made the same error in 8 tool calls (31s). Judge accepted the false claim at 0.98 confidence. A single `ls hooks/` would have answered correctly in 0.1s. Meanwhile, Gemini 503'd for 18 min through 4 retries. | For verifiable claims (file exists, function present, config value), use deterministic checks (grep, ls, test hooks) not multi-model debate. Consider removing --enable-tools from /challenge — the tool calls produced false confidence, not better results. debate.py's value is subjective judgment (scope, design, architecture), not fact verification. |
| L36 | Zero-functionality refactors of a monolith are safe when the test suite is fast and commits stay small | debate.py split this session: 4,629 → 4,369 lines across 4 commits, each extracting one subcommand. Pattern: function body moves to `scripts/debate_<topic>.py` verbatim; debate.py re-imports the name; a lazy `import debate` inside the new module pulls shared state. 923 tests stayed green between every commit (7s suite enabled fast verification). Zero behavior change by construction — no logic moved relative to callers. | Fast test suite (sub-10s) + small commits + lazy imports for shared state = monolith extraction without risk. Don't need a full package restructure; sibling modules work if the monolith is already the shared-state module. The "cannot lose any functionality" constraint is met by construction — extraction is a rename, not a refactor. |
| L37 | Rules that depend on author discipline keep failing — enforcement has to live in a single owned surface, not in prose | `/start` dumped raw JSON from 4 diagnostic scripts at every session start even though `.claude/rules/skill-authoring.md` already had a "No Framework Plumbing" rule. The rule caught violations only when the user complained. Fix was structural: scripts go silent-on-success, `scripts/bootstrap_diagnostics.py` is the one file that owns the user-visible output surface. New diagnostics register with the wrapper instead of adding Bash calls to the skill — authors physically can't leak plumbing because they don't touch stdout. | When a class of violation recurs despite an existing rule, escalate on the enforcement ladder: move the behavior into a single-owner artifact (wrapper, hook, codegen, generated file) so "leaking" stops being a thing authors can accidentally do. Prose rules work for one-offs; recurring patterns need structural ownership. |
| L39 | Monolith extractions surface latent module-identity splits when test files manipulate `sys.modules` — refines L36 with the boundary condition | debate.py split 6/N (cmd_explore extraction). 3 test files (test_debate_posture_floor.py, test_debate_pure.py, test_debate_utils.py) had a module-level `for mod in sys.modules: del sys.modules[mod]; import debate` block. At pytest collection time this created two distinct `debate` module objects. `fake_credentials` patched module A (test_debate_commands.py's `debate` reference). The extracted `cmd_explore`'s lazy `import debate` resolved to module B from sys.modules → unpatched → 3 `TestCmdExplore` tests failed in the full suite, passed in isolation. Before the extraction, `cmd_explore` lived inside module A; its `__globals__` lookup pointed at A regardless of what sys.modules contained. The lazy-import pattern's "zero behavior change" property held only because no one was mutating sys.modules between collection and test run. Prior session's diagnosis ("pre-existing test-ordering pollution, not extraction regression") was wrong — the extraction is what surfaced the bug. Fixed by removing the cargo-cult sys.modules block from the 3 test files (commit c993edf), then committing the split. | L36's "safe by construction" claim is correct only when no test code manipulates `sys.modules`. Before any monolith extraction: `grep -rn 'sys\.modules' tests/`. If matches exist, fix them first (they're almost always cargo-cult — a plain `import debate` exposes everything). After extracting, run the FULL suite (not just the file-under-test) at least once to catch lazy-import vs. fixture-patching gaps. Module-identity splits are invisible at code-review time — they only surface when call-time lookup meets cross-module monkeypatching. |
| L42 | Ad-hoc per-call-site fixes for LLM param deprecations create silent coverage gaps — centralize via a sanitizer + contract test | Opus 4.7 `temperature` deprecation: commit 92d2e3e patched 2 of 4 dispatch paths in `scripts/llm_client.py` (`_sdk_call`, `_legacy_call`). The other 2 paths (`_anthropic_call`, `_sdk_tool_call`) still sent the param; `/review` and `/challenge --enable-tools` both route through `_sdk_tool_call` and 400'd on every Opus 4.7 challenger for days without anyone noticing until a review was actually run. Per-site string checks (`if not model.startswith("claude-opus-4-7")`) also don't cover variants like `claude-opus-4-7-1m` or `anthropic/claude-opus-4-7`. | Replaced in commit 750fe9b with `MODEL_DEPRECATED_PARAMS` capability table + `_sanitize_llm_kwargs()` called from every dispatch path. Contract test `test_all_llm_paths_route_through_sanitizer` greps the module; fails if any dispatch function stops calling the sanitizer. Pattern: when provider deprecations hit ≥2 call sites, refactor to a single source of truth rather than duplicating conditionals. The regression surface is the set of call sites; the fix surface must match. |
| L44 | N=1 is not data — LLM tuning decisions must ground in paired output-quality comparisons across n≥3 (ideally 5+) representative inputs. Quality first, speed/cost second. | Frame-lens development (2026-04-17) almost shipped a cross-family model change (frame-factual: sonnet → gpt-5.4) based on a single gbrain run showing 3x speedup and apparent finding diversity. One data point is consistent with any story — model difference, prompt difference, proposal difficulty, randomness. Corrected by running the same pair across n=5 historical proposals: cross-family was BETTER on 4/5 (more architectural findings, higher precision per finding, better synthesis of staleness implications) and TIED on 1/5. The 3x speedup turned out to be real but secondary — leading with speed was the wrong frame. Same pattern earlier in the session: "tools bias frame critique" was formed on n=1 (gbrain tools-on vs tools-off) and only became credible after n=4 paired validation. | Before shipping any LLM tuning decision (model swap, prompt edit, threshold change), the evidence bar is paired comparisons on n≥3 (ideally 5+) representative inputs, scored on **output quality** — not on speed, cost, or token count. Speed/cost are secondary considerations once quality is established. Quality dimensions to compare: precision per finding (signal-to-noise), recall on known-defect proposals, verdict alignment with ground truth, and whether the change catches a *kind* of issue the prior config systematically missed. "It was faster" or "it produced more findings" are not quality measures. The asymmetry: 5 parallel calls cost ~30-60s and pennies; shipping a bad tuning decision (or a good one for the wrong reason) costs hours of debugging. Applies to: model selection, persona prompt changes, tool-access defaults, temperature adjustments, threshold tuning. |
| L45 | /think discover drifts off-scope when user mentions evaluative criteria mid-conversation — must re-read handoff, not mutate scope from conversation | 2026-04-17 session: user asked "/think discover on applying L43/L44 paired-audit method to other personas and systems" (handoff named this unambiguously). Mid-conversation user added "It's only worth it IF it produces better results that save time and money long-term." I read this as a scope reframe from "generalize Frame's dual-mode" to "audit whole debate system's ROI" and wrote a full ROI-audit design doc + Phase 0 feasibility + labeling rubric. None of it addressed the original ask. User corrected after I'd already committed 1c40d6c. Cost: ~1hr of session + one misleading commit in history + context bloat. The drift happened because the evaluative language ("worth it", "save money", "better results") is ambient criterion language — it names WHY a generalization matters, not WHAT to do instead of generalizing. I treated it as pivot rather than motivation. | When a user states evaluative criteria during a discovery session, check: does the criterion CONTRADICT the scoped question, or does it name the motivation for the scoped question? If ambiguous, pause and re-read the original handoff before designing. Specifically for /think discover: before Phase 4 (alternatives generation), re-read `tasks/handoff.md` "Next Session Should" section verbatim. If the design doc's scope no longer matches the handoff scope, that's drift — stop and check with user. Evaluative language like "worth it / ROI / value / better" almost always names the criterion, not a new scope. |
| L43 | Tool access biases frame critique toward positive-space enumeration — structural frame reasoning requires tools-off; factual claim verification requires tools-on. Running both as dual-mode catches distinct failure classes. Refines L33. | Gbrain-adoption debate (2026-04-16) passed 3-persona `/challenge` review but missed the real answer (compositional SQLite+gbrain-shaped TEXT column) — the candidate set was framed as binary. Frame-lens paired validation (n=4 proposals, 2026-04-17): tools-on catches factual contradictions that tools-off physically cannot — e.g., "proposal argues to ship commits `6a3fb0b`/`62e317b` that already shipped"; "`refine_rotation` has zero Claude models, contradicting the 'Claude-only refine' motivation"; "3-model-family-diversity claim contradicted by config (2 Claude + 2 Gemini + 1 GPT)". Tools-off catches structural reasoning gaps that tools-on misses — e.g., "problem is a low-frequency inconvenience inflated to justify two-phase architecture"; "close the existing `[TRIVIAL]` bypass instead of adding a new hook"; "Phase 2 routing layer is over-engineered for a single integration point". On every proposal, ≥2 MATERIAL findings were mode-exclusive. Tools-off was 2.4x faster (~30s vs ~75s, 0 vs ~25 tool calls) and produced equal or more MATERIAL findings (3/3/3/3 vs 2/2/3/2). Related: source-driven proposals inherit the source's frame — when an agent composes a proposal by fetching external content (a repo, library, paper), the candidate set tends outward-in (what's in the source) rather than problem-inward (what solves the problem). Cross-family dual-mode (structural=claude-sonnet-4-6, factual=gpt-5.4) reduces correlation between halves and runs ~3x faster (GPT terminates tool loops in ~17 calls vs Sonnet's ~56). | Frame persona in `scripts/debate.py` expands to dual-mode when `--enable-tools` is on: `frame-structural` (tools forcibly off, reasons from proposal text alone) runs in parallel with `frame-factual` (tools on, verifies proposal claims against codebase). Factual half routes to a different model family (config `frame_factual_model`, default `gpt-5.4`) for diversity. When `--enable-tools` is off, only `frame-structural` runs. The judge stage dedups overlapping findings. **Validation hygiene:** when calibrating the frame lens, use proposals whose failure modes are NOT documented in `docs/current-state.md`, `tasks/lessons.md`, or session wraps — or hold those files out of the model's tool scope during the test. A tool-enabled lens that reads its own retrospective is citing the answer, not deriving it. The production use case (fresh proposal, no pre-written post-mortem) is unaffected, but "passed validation" against a self-documented failure is weaker evidence than "passed validation from scratch." **Generalization (2026-04-17, n=5 per persona across 5 historical proposals — autobuild, explore-intake, learning-velocity, streamline-rules, litellm-fallback):** architect ADOPTS dual-mode (5/5 BIDIRECTIONAL — tools-on catches "already-shipped" on 4/5 and code-grounded architectural drift; tools-off catches structural/rollback gaps). Security DECLINES at pre-committed 5/5 threshold (4/5 BIDIRECTIONAL — passes on 4 proposals, fails on explore-intake where tools-off collapsed 2 findings into the same prompt-injection defect while tools-on added exfiltration + wrong-integration-point findings). PM DECLINES (2/5 BIDIRECTIONAL — PM without tools captures most of its value via prompt-engineering/UX/sequencing critiques; tools-on exclusive findings are rare). **Pattern:** dual-mode benefit correlates with how much a persona's value comes from codebase-verification vs prompt-level reasoning. Evidence: `tasks/dual-mode-generalization-results.md` + three per-persona analysis files. Security's 4/5 is evidence a lower threshold could justify dual-mode for security — not adopted here; would require new pre-committed threshold + re-validation. **Frame-reach intake audit (2026-04-17, n=5 audit proposals + 1 negative control):** tested whether running dual-mode Frame at proposal intake (before `/challenge` fires) reliably reproduces Frame-inside-challenge findings. VERDICT: DECLINE. Three locked threshold gates, two failed: (1) per-proposal reproduction ≥70% failed on autobuild (60%) and learning-velocity (40%) under strict MATERIAL-only rubric — several Round 2 MATERIAL findings showed up as ADVISORY in intake, i.e., severity-level drift across runs with identical prompts; (2) the litellm-fallback "already-shipped" binary gate PASSED (frame-factual reproduced it cleanly with code citations); (3) negative control produced 2 MATERIAL findings when 0 were required — frame-factual correctly flagged stale code claims in the synthetic "clean" proposal, proving that "frame-clean ≠ factual-claim-free" and invalidating the binary 0-MATERIAL heuristic as an FP bound. Compound implication: intake-gate integration cannot rely on MATERIAL count as a reject/proceed signal — the signal is noisier than Round 2 implied due to severity drift AND factual verification always surfaces findings on code-citing proposals. **Status of intake-check subcommand:** `debate.py intake-check` exists as a composable primitive but is NOT wired into `/challenge` as a pre-check. Judge/refine/premortem reach audits deferred pending a different integration design that accounts for severity drift + factual-FP asymmetry. Evidence: `tasks/frame-reach-intake-audit-results.md` + intake runs in `tasks/frame-reach-intake-audit-runs/`. **Judge-stage Frame-reach audit (2026-04-17 evening, Phase 0x + Phase 0a only):** VERDICT: DECLINE — pre-committed gate (0-1 missed_disqualifier errors → DECLINE) fired at 1/5. The single error (litellm-fallback) was a corpus-measurement misalignment artifact, not a judge defect: Round 2 ground truth for that proposal = REJECT, but the REJECT depended on Frame's already-shipped finding, which is absent from the 3-persona challenger input that baseline cmd_judge receives. Baseline judge produced REVISE — the correct verdict given its input. The other 4 proposals (ground truth REVISE) cannot produce missed-disqualifier errors under any judge behavior. **Core learning:** Corpus-ground-truth alignment is a prerequisite for judge-stage audits. A corpus with Frame-dependent ground truth paired with Frame-absent judge input produces systematic readings that don't measure judge quality. Future judge-stage audits need either (a) corpus where ground truth is derivable from challenger findings alone, or (b) measurement design that feeds Frame-augmented findings to judge. **Reusable artifact:** `scripts/judge_frame_orchestrator.py` ships as a standalone harness — dual-mode Frame (gemini structural + gpt factual) post-judge critique with family-overlap enforcement + credential-pattern pre-check. Sunk cost is not wasted; any future corpus can reuse the harness. Evidence: `tasks/judge-stage-frame-reach-audit-results.md`. |
