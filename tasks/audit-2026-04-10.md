## Findings

### F1: audit.db is empty but documentation and code assume 18,500+ rows
- **Severity:** critical
- **Source:** inspected
- **Evidence:** `stores/audit.db` is 0 bytes (created 2026-04-10). `.claude/rules/reference/operational-context.md` describes the `audit_log` table as having "18,500+ rows" and provides 3 fixed queries. `scripts/debate_tools.py:134-145` (`_get_recent_costs`) reads from it. `.claude/skills/status/SKILL.md` queries it for recent activity. `.claude/skills/challenge/SKILL.md` pulls operational context from it.
- **Impact:** `/status` skill routing, cost tracking, and operational health checks all silently return empty results. The system believes it has no history.
- **Not checked:** Whether audit.db was populated in the downstream Jarvis project and the reference was carried over without the data.
- **Recommendation:** Either (a) implement audit logging writes in BuildOS scripts (debate.py, hook runs, skill invocations), or (b) remove the 18,500+ row claims from docs and mark audit.db as an optional downstream feature. The current state is a silent failure — code runs, gets nothing, reports "clear day."

### F2: Contract tests all fail — they reference outbox_tool.py which doesn't exist
- **Severity:** critical
- **Source:** inspected
- **Evidence:** All 7 contract test files in `tests/contracts/` depend on `scripts/outbox_tool.py` via the `$TOOL` variable set in `tests/contracts/helpers.sh`. This file does not exist in the repo. Results: `test_approval_gating.sh` (0/7 pass), `test_degraded_mode.sh` (1/12), `test_idempotency.sh` (2/6), `test_exactly_once_scheduling.sh` (2/10), `test_rollback.sh` (0/12), `test_state_machine.sh` (0/13), `test_version_pinning.sh` (2/2 — different failure: missing `docker/docker-compose.yml`).
- **Impact:** The "Essential Eight" invariants — idempotency, approval gating, audit completeness, degraded mode, state machine validation, rollback, version pinning, exactly-once scheduling — are listed as hard rules in CLAUDE.md but are not verified by any passing test.
- **Not checked:** Whether these tests pass in a downstream project that has outbox_tool.py.
- **Recommendation:** Either (a) rewrite contract tests to exercise BuildOS's own code paths (debate.py, hooks, skill execution), or (b) mark them as downstream-project templates and add BuildOS-native contract tests. Tests that exist but can't pass are worse than no tests — they create a false sense of coverage.

### F6: SQL identifier interpolation in debate_tools.py
- **Severity:** warning
- **Source:** inspected
- **Evidence:** `scripts/debate_tools.py:94` — `f"SELECT COUNT(*) FROM {table} WHERE {col} = ?"`. While `table` and `col` are checked against `ALLOWED_TABLES` and `ALLOWED_COLUMNS` whitelists before interpolation, the identifiers are still f-string interpolated into SQL. The value parameter is properly bound with `?`.
- **Impact:** Low in practice (whitelist prevents injection), but violates the project's own security rules about untrusted data in query construction. If the whitelist dictionaries are ever expanded carelessly, the surface opens.
- **Not checked:** Whether any code path can bypass the whitelist check.
- **Recommendation:** Wrap table/column names in double-quotes in the SQL string (`f'SELECT COUNT(*) FROM "{table}" WHERE "{col}" = ?'`) as defense-in-depth, or use an explicit mapping from allowed names to query fragments.
