# Design: Multi-Model Skills ROI Audit

Generated by /think discover on 2026-04-17
Branch: main
Mode: Builder (internal infrastructure ROI)

## Problem Statement

We have spent substantial engineering effort building the cross-model debate system and the six skills that depend on it: `/challenge`, `/review`, `/polish`, `/explore`, `/pressure-test`, and the judge+refine rounds inside `/challenge --deep`. The system costs real money (per-run API spend) and real time (60-120s per invocation), and imposes cognitive load on the reader who must triage findings.

We have **no systematic evidence** that this infrastructure earns its cost. The evidence we do have is mixed: frame-lens-validation (n=5) showed panel-of-4 beat panel-of-3 with ~30 novel MATERIAL findings and one verdict flip; but L28 showed 3 models unanimous-and-wrong, L33 showed `--enable-tools` produced 0.98-confidence false claims, and D5 showed single-model + divergence prompts beat multi-model for thinking tasks. Debate value is clearly non-zero, but also clearly non-uniform across task shapes.

The load-bearing question is not "is debate 'worth it'?" (unanswerable in the abstract). It is: **for each multi-model skill, is the marginal value of multi-model infrastructure worth the marginal cost — and if so, what's the right tuning?**

## What Makes This Worth Doing

- **Decision-relevant.** Verdicts per skill directly change framework behavior: keep, tune, or drop. A drop verdict for even one high-cost skill (e.g., `/polish` with 6 rounds × 3 models) cuts cost meaningfully.
- **Infrastructure reusable.** The paired-replay harness built in Phase 2 operationalizes L44 as durable infrastructure — every future tuning decision (new persona, model swap, prompt edit, tool-posture change) gets paired-validated by the same workflow. L44 stops being a written rule and becomes a tool.
- **Absorbs existing work.** The `debate-efficacy-study-plan.md` that's already on disk (unshipped) becomes a Phase 2 sub-experiment for `/challenge` if Phase 1 keeps it. No wasted work; no ambiguity about what to do with the prior chain.
- **Answers the meta-question honestly.** If our multi-model infrastructure is theater, we should find out — cheaply — and simplify. If it's load-bearing, we should tune it deliberately. "Vibes and localized wins" is not an acceptable justification for framework spend at this scale.

## Premises

1. **Finding value is labelable retrospectively.** For each MATERIAL finding a skill produced, we can classify it as one of: (a) prevented a real problem, (b) meaningfully changed the plan, (c) noise/ignored, (d) actively wrong (cost reader time, prevented nothing). Evidence: git log + tasks/decisions.md + session-log entries + shipped-commit diffs. Not perfect — some findings shape future thinking without triggering an immediate change — but usable at the decision-relevant resolution.
2. **Cost is easy; value is the labor.** `stores/debate-log.jsonl` + file timestamps give us per-run API cost + wall time. Human labeling of findings-to-outcomes is the labor-heavy step.
3. **ROI is per-skill, not aggregate.** Each multi-model skill has a different task shape, different finding population, different failure modes. Expect verdicts like "keep `/challenge`, drop `/polish`, tune `/review`." A single "multi-model = worth it / not" verdict would hide the decision-relevant differences.
4. **The existing `debate-efficacy-study` plan is a Phase 2 sub-experiment.** Its C-vs-D comparison is a tuning question for `/challenge`. Only meaningful if Phase 1 keeps `/challenge`. Absorbed here; not run independently.
5. **Absence of value is evidence.** A skill that routinely produces thin MATERIAL findings on historically-important work is the strongest drop candidate. "It didn't catch anything important" is a legitimate verdict, not a data quality problem.
6. **Phase 2 operationalizes L44 as durable infrastructure.** Any proposed tuning change for a keep-verdict skill replays against N historical inputs, old vs new config, ROI-delta measured. The harness is reusable across skills and tuning changes.

## Approaches Considered

### Approach A: Minimal sequential — `/challenge` first, replicate (CHOSEN)

Phase 1 audits `/challenge` ROI only. If verdict = keep or tune, run existing `debate-efficacy-study-plan.md` as first Phase 2 experiment. After `/challenge` audit method proves out, replicate to other skills in sequence: `/review`, then `/polish`, then `/explore`, then `/pressure-test`, and the judge+refine sub-rounds. Learnings from each audit inform the next.

- **Effort:** S per skill (~2-3 days Phase 1 audit + conditional Phase 2), sequential across ~5-6 skills
- **Risk:** Low — rubric battle-tested on richest log first
- **Pros:** Cheapest path to first answer; rubric improves across skills; early "drop" verdicts kill downstream work; existing plan absorbed cleanly
- **Cons:** Other skills wait for their turn; risk of inconsistent rubric across time
- **Reuses:** frame-lens-validation methodology; `stores/debate-log.jsonl`; `tasks/decisions.md`; existing `debate-efficacy-study-plan.md`

### Approach B: Full parallel sweep

Phase 1 audits all 6 multi-model skills in parallel. Per-skill verdict. Phase 2 conditional per skill.

- **Effort:** M (1-2 weeks P1 + conditional P2)
- **Risk:** Med — rubric designed against 6 skills simultaneously is harder to get right; may discover some skills have too-sparse logs and we've over-committed
- **Pros:** One pass gives the matrix; clean cross-skill comparison
- **Cons:** Labor-heavy upfront; no early-drop short-circuit

### Approach C: Instrumented continuous

Build outcome-capture into every multi-model skill invocation (extend `stores/debate-log.jsonl` schema). Manual backfill of history. Going forward, every run auto-captures "was finding acted on / did action matter." Monthly ROI reports.

- **Effort:** L (infra + labeling + ongoing maintenance)
- **Risk:** Premature infrastructure before we know keep/drop verdict
- **Pros:** Future tuning decisions data-driven by default
- **Cons:** Heavy investment before answering the prerequisite question

### Approach D: No study, decide from heuristics

Write up current best judgment (L28/L33/D5 contra; frame-lens pro) as a decision. Ship the verdict, skip the study.

- **Effort:** XS (hours)
- **Risk:** High — no direct data; decision may be wrong
- **Pros:** Fastest
- **Cons:** The whole point of this program is to escape vibes-based justification for framework spend. D reproduces the problem.

## Recommended Approach: A, with explicit "extend to all skills after /challenge learnings" phasing

### Phase 0 — Feasibility check (30 min)

Before committing full Phase 1 labor, audit `stores/debate-log.jsonl` for `/challenge` entries. Count: total `/challenge` runs with MATERIAL findings, how many findings total, how many have outcome-traceable references (commit hashes, decision numbers, plan files). If density is too low (<20 findings with traceable outcomes), switch to a wider scope for Phase 1 (include `/review` entries). Kill the study entirely if total labelable findings across all skills is <30 — at that point we lack evidence even for retrospective labeling and should either wait for more data or go to Approach D.

### Phase 1 — ROI audit for `/challenge`

1. **Build the labeling rubric.** Four categories (P1 above) + a "mixed/unclear" escape. Pilot rubric on 5 findings, iterate, freeze. Rubric document lives in `tasks/multi-model-skills-roi-audit-rubric.md`.
2. **Enumerate `/challenge` findings from history.** Grep `stores/debate-log.jsonl` for `/challenge` phase entries; cross-reference with `tasks/*-challenge.md` and `tasks/*-judgment.md` artifacts on disk. Target: last ~20 invocations or ~100 MATERIAL findings, whichever comes first.
3. **Label each finding.** Walk each finding → find the commit/decision that acted on it (or evidence it was dismissed). Label (a)/(b)/(c)/(d). Justin does the labeling; Claude does the mechanical cross-referencing and presents the evidence for each.
4. **Compute ROI.** Per invocation: cost = API $ + wall time * hourly-rate; value = sum of finding weights (a=+3, b=+1, c=0, d=-1 is a starting weighting — we'll revisit after labeling pilot). Aggregate across invocations. Sanity-check distribution: are most findings in (c) / (d)? That's the drop signal.
5. **Emit verdict.** Keep / tune / drop for `/challenge`. If keep: note open tuning questions (cross-family vs Claude-only? tool posture per persona?).
6. **Output:** `tasks/multi-model-skills-roi-audit-challenge-results.md` with raw findings table, per-invocation ROI, verdict, evidence.

### Phase 2 (conditional on keep/tune verdict) — Paired replay infrastructure for `/challenge`

If verdict is keep or tune, run the existing `debate-efficacy-study-plan.md` as the first paired tuning experiment. It was already designed, already has challenge findings incorporated, already has fixes F1-F6 in the plan. Execute it. Output goes into this broader program as `tasks/multi-model-skills-roi-audit-challenge-tuning-c-vs-d.md`.

Generalize the harness it builds (two-arm paired replay across 5 historical inputs, judge, dual-label) into `scripts/paired_replay.py` (or similar) so future tuning experiments don't re-plumb.

### Phase 3 — Replicate to other multi-model skills

Applied in this order (most expensive / most frequently invoked first):

1. `/review` — touches every non-trivial commit; 3 lenses × 3 models. If drop, huge cost savings.
2. `/polish` — 6 rounds × 3 models is the most expensive single skill per invocation. Hypothesis: diminishing returns past round 3 — a Phase 2 tuning question.
3. `/pressure-test` — lower frequency, specialized. Audit last.
4. `/explore` — may not have enough history for Phase 1 — consider Approach C instrumentation if so.
5. `/challenge --deep` judge+refine sub-rounds — only if `/challenge` base audit is "keep."

Each follows the same Phase 1 → Phase 2 structure, with the rubric frozen from `/challenge` learnings.

## Open Questions

1. **Labeling subjectivity.** Will Justin's labels be reproducible? Dual-labeling 10-20% of findings with a second pass (after time gap) gives an agreement rate. If <80%, the rubric needs sharpening.
2. **What's a "real problem prevented"?** Some findings shape future thinking without triggering a specific commit. Class (a) needs a concrete definition — probably "commit exists that implements or avoids something the finding named, and the fix would have been costly or blocking otherwise." Needs pilot iteration.
3. **Weighting of (a)/(b)/(c)/(d).** The +3/+1/0/-1 scheme is a starting point; should be revisited after first-pass labeling reveals distribution shape. Sensitivity analysis: does the verdict change under different weightings?
4. **Cognitive-load cost.** Not captured in API $ + wall time. Proxy via "findings per invocation" × "reader minutes per finding"? Defer until we see labeling data.
5. **How many invocations is enough?** n=20 is starting point. May need more for low-(a)-rate skills to distinguish "rare but valuable" from "never valuable."
6. **Should we skip the Phase 5.5 spec review panel on this design?** Ironic to audit debate via /review and /challenge of the audit itself. Probably worth running the meta-panel as its own data point on /review's value.

## Known Limits

- **Retrospective contamination.** Any Claude instance labeling findings has seen those proposals and their outcomes in training or context. Labels are informed, not blind. Dual-label + Justin as ground truth reduces but doesn't eliminate this.
- **Counterfactual ambiguity.** "What would have happened without debate?" is not directly observable. We approximate via (c)/(d) rate — if most findings are ignored or wrong, the counterfactual is "not much different."
- **Tuning floor.** Even if /challenge ROI is low, it may be because we're running it wrong, not because the idea is wrong. Phase 2 tuning can recover skills Phase 1 was about to kill. Don't be hasty on the drop verdict before one tuning pass.
- **Long-tail value.** A single caught-in-time bug that prevents an incident can be worth hundreds of noisy runs. Small-n retrospective labeling can miss this. Phase 2 replay + forward instrumentation (Approach C, later) addresses it.
- **Scope creep risk.** Every skill we audit is a week. If we're not careful, this audit program becomes its own overhead problem. Cap: 6 skills over 6-8 weeks, or kill the program and go to Approach D.

## Dependencies

- `stores/debate-log.jsonl` must have enough outcome-traceable entries for `/challenge`. Phase 0 verifies.
- Existing `debate-efficacy-study-plan.md` stays on disk; we reference it in Phase 2, don't replan.
- LLM judge availability for Phase 2 (Gemini 3.1 Pro or equivalent held-out from audited arms).
