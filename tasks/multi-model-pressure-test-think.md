# Brief: Multi-Model Pressure-Test (D5 Phase 2)

Generated by /think refine on 2026-04-15

## Problem Statement
`cmd_pressure_test` is single-model (D5 decision). The decision foundation audit found this doesn't hold for adversarial critique — reasoning independence requires cross-family diversity that a single model can't provide. The original benchmark may have been context-corrupted. The audit recommends A/B testing single vs multi-model specifically for pressure-test.

## Proposed Approach
Add `--models` flag to `cmd_pressure_test` parser. When multiple models are specified, run them in parallel via ThreadPoolExecutor (same pattern as review panel at line 3424), anonymize and position-randomize results (Zheng et al., same as review), then run a synthesis step using a model from a different family than any of the inputs.

Key design points:
- `--model` (existing, single) and `--models` (new, multi) are mutually exclusive
- When `--models` used: each model gets the same system prompt and proposal, runs independently
- Synthesis model defaults to config `judge_default` (gpt-5.4) — different family from typical pressure-test models
- Output format: individual analyses (labeled A/B/C) + synthesis section
- Fallback: each model gets `_call_with_model_fallback` (D18 already wired)
- Minimum 2 models required for multi-model mode; single `--model` keeps existing behavior

## Key Assumptions
- ThreadPoolExecutor parallel pattern is proven (used in challenge, verdict, review panel — 4 existing sites)
- Synthesis step adds value by reconciling disagreements across models — not just concatenation
- The pressure-test system prompt works across model families without modification
- D18 fallback wiring (already shipped) handles individual model timeouts gracefully

## Risks
- Synthesis model may just average findings instead of identifying genuine disagreements — mitigated by explicit prompt design ("where do the analyses DISAGREE?")
- 3x cost for multi-model runs — acceptable since pressure-test is infrequent (pre-plan gate, not per-commit)
- If all models agree, multi-model adds cost without insight — but that's signal too (high-confidence finding)

## Simplest Testable Version
1. Add `--models` to pressure-test parser (mutually exclusive with `--model`)
2. Parallel execution via ThreadPoolExecutor (copy review panel pattern)
3. Position-randomized anonymous labeling (A/B/C)
4. Synthesis step: one LLM call with all outputs + "where do analyses disagree?" prompt
5. Output: frontmatter + individual sections + synthesis section
6. SKILL.md update to document `--models` option

No changes to premortem frame in v1 — multi-model premortem is a future enhancement if multi-model pressure-test proves valuable.
