# Design: BuildOS Upstream Propagation

Generated by /define discover on 2026-03-30
Branch: main
Status: DRAFT
Mode: Product

## Problem Statement

BuildOS is a shared framework (rules, hooks, skills, scripts, CLAUDE.md) distributed via GitHub. Improvements are discovered during daily production work in OpenClaw (the downstream project), but don't flow back to the BuildOS repo. OpenClaw is ~4 months ahead. Other users pull BuildOS from GitHub but consumption patterns aren't established yet.

The core tension: refinements happen in project context (where they're battle-tested), but must reach the framework repo (where consumers pull from) in generalized form. Two compounding problems block this: (1) generalization is hard — stripping project-specific references is tedious and error-prone, and (2) the boundary between framework and project code is blurry.

## Demand Evidence

- Users actively pull from GitHub for updates
- A detailed 5-batch sync plan exists but hasn't been executed — evidence the problem is real and the manual process doesn't scale
- OpenClaw's production usage generates continuous framework improvements (hooks, rules, skill patterns) that are lost to consumers

## Status Quo

Current workflow: discover improvement in OpenClaw → improvement stays in OpenClaw → periodically plan a sync session → manually diff, generalize, copy to claude-build-os → push. In practice, the sync sessions don't happen because generalization is tedious and the boundary is unclear.

Cost: every consumer is running a framework that's months behind what's been proven in production. New projects start from a stale baseline.

## Target User & Narrowest Wedge

**Target user:** Engineer who adopted BuildOS for their Claude Code project and wants ongoing governance improvements without rebuilding from scratch.

**Narrowest wedge:** A structural separation in OpenClaw that makes framework files generalization-free, plus a simple sync script that copies them to claude-build-os.

## Premises

1. BuildOS files must live in the project directory — Claude Code reads .claude/rules/, CLAUDE.md from the working tree (AGREED)
2. Users want ongoing framework updates, not a one-time scaffold (AGREED — but challenged by reviewer; needs validation as consumption patterns emerge)
3. Generalization (stripping project references) is unavoidable in the current architecture (AGREED — but Approach A eliminates this by making the boundary structural)
4. The two-repo model's main cost is that improvements don't flow upstream, not that two repos exist (AGREED)

## Independent Perspective

**Steelman:** Building an automated extraction pipeline that identifies generalized improvements in downstream projects, strips proprietary context, and pushes upstream to a shared framework repo.

**Key insight from reviewer:** "The boundary between framework and project code is blurry" reveals that building a sync tool is currently impossible. The creator can't cleanly articulate the boundary, so automation can't either. Must redesign the architecture so the boundary is structurally enforced, not vibes-based.

**Challenged premise:** "Users want ongoing framework updates" — reviewer argues that when framework files live in the working directory, users modify them, and upstream updates cause merge conflicts. Need to validate whether consumers keep framework files pristine or customize them.

**48-hour prototype suggestion:** Use an LLM to generalize files instead of regex transforms. Interesting but fragile for production use.

## Approaches Considered

### Approach A: Layered Architecture (Structural Boundary) — RECOMMENDED

Keep two repos. Redesign OpenClaw so framework files and project files are structurally separated by directory, not convention.

**Framework layer** (generic, syncs to BuildOS):
- `.claude/rules/` — 7 base rule files (code-quality, design, orchestration, etc.)
- `.claude/skills/` — 13+ framework skills (challenge, plan, define, review, etc.)
- `scripts/` — framework scripts (debate.py, tier_classify.py, etc.)
- `hooks/` — 11 enforcement hooks
- `CLAUDE.md` — template with project-specific section clearly marked

**Project layer** (OpenClaw-specific, never syncs):
- `.claude/rules/reference/` — 6 project-specific rule files (already separated!)
- `skills/` — 4 production skills (email-triage, etc.)
- `scripts/` — project toolbelt (*_tool.py, deploy_*, health_daemon)
- `primitives/`, `jarvis-app/`, `stores/`

**Sync mechanism:** A `buildos-sync.sh` script that:
1. Diffs framework-layer files between OpenClaw and claude-build-os
2. Shows what changed (file list + summary)
3. Copies changed files to claude-build-os
4. Runs the security scan (grep for project-specific references — should find zero)
5. Commits and pushes (or creates a PR for review)

**Key principle:** Framework files never contain project-specific references. If a framework rule needs project-specific behavior, the project adds an override in the project layer. This is how `.claude/rules/reference/` already works — it's the extension point.

Effort: M | Risk: Low
Pros: Clean boundary (structural, not vibes), no generalization transforms needed, users get clean upstream pulls, already 80% there with reference/ pattern
Cons: Requires one-time refactoring to move project refs out of framework files, two repos still exist (but that's fine)

### Approach B: Single Repo + Published Extract

OpenClaw becomes sole source of truth. Auto-extract framework files via CI/hook, apply regex generalization, force-push to claude-build-os.

Effort: S | Risk: Med
Pros: Single source of truth, automated
Cons: Regex generalization is fragile and the exact problem today, consumers get generated artifact not curated product

### Approach C: Copier Template + Upstream-First

Convert claude-build-os to a Copier template. Adopt upstream-first discipline.

Effort: L | Risk: Med
Pros: Industry-standard, bidirectional sync
Cons: Reverses natural workflow (improvements happen in project, not framework), Copier learning curve, merge conflicts

## Recommended Approach

**Approach A: Layered Architecture.**

Rationale:
1. **Solves the root cause** — the blurry boundary becomes a structural boundary. No more "is this framework or project?"
2. **Eliminates generalization** — if framework files are clean, sync is just copy. No regex, no LLM rewriting.
3. **Already 80% there** — the `.claude/rules/reference/` pattern IS the project override layer. Extend it to skills and scripts.
4. **Matches natural workflow** — discover in project, refactor to keep framework files clean, sync to BuildOS. No upstream-first discipline reversal.
5. **Two repos is fine** — the cost isn't two repos; it's the generalization friction. Remove the friction, the two-repo model works.

## Open Questions

1. **Consumer customization:** Do BuildOS consumers modify framework files directly, or keep them pristine and add overrides? This determines whether `copier update`-style conflict resolution is needed later.
2. **CLAUDE.md split:** CLAUDE.md currently mixes framework rules and project-specific config. Should it be split into `CLAUDE.md` (framework) + `CLAUDE.project.md` (project)? Claude Code only auto-loads CLAUDE.md — project rules could go in `.claude/rules/reference/` instead.
3. **scripts/ boundary:** Some scripts serve both roles (e.g., `debate.py` is framework but has hardcoded model names). Need to audit which scripts are truly generic.
4. **BuildOS-only skills:** BuildOS has skills not in OpenClaw (audit, capture, handoff, setup, sync, triage). Where do improvements to those come from? They need their own development path.
5. **Sync frequency:** Session-close hook? Weekly batch? On-demand? Start with on-demand, add automation after the boundary is clean.

## Success Criteria

1. A framework improvement in OpenClaw reaches claude-build-os within the same session (or next session close)
2. The security scan (grep for project references in framework files) returns zero results at all times
3. `buildos-sync.sh` runs in <30 seconds and requires zero manual generalization
4. A new OpenClaw project bootstrapped from claude-build-os gets all production-proven framework improvements

## Distribution Plan

Consumers continue to `git clone` or `git pull` from claude-build-os on GitHub. No change to their workflow. The improvement is entirely on the push side.

## Reviewer Findings (Round 1)

Scores: 10/10, 8/10, 9.5/10 (avg 9.2). Key issues from Reviewer B, fixed below:

### Fixed: Ownership & Conflict Policy
- **OpenClaw framework layer is canonical** for all shared files. BuildOS-only skills (audit, capture, handoff, setup, sync, triage) are maintained in claude-build-os directly.
- If both repos changed the same framework file, sync script aborts and requires manual review.

### Fixed: Directory Structure Clarification
| Layer | Path | Syncs to BuildOS? |
|-------|------|-------------------|
| Framework rules | `.claude/rules/*.md` (excl. reference/) | Yes |
| Framework skills | `.claude/skills/*/SKILL.md` | Yes |
| Framework scripts | `scripts/debate.py`, `tier_classify.py`, `recall_search.py`, `finding_tracker.py`, `enrich_context.py`, `artifact_check.py` | Yes |
| Framework hooks | `hooks/hook-*.sh`, `hooks/hook-*.py` | Yes |
| Project rules | `.claude/rules/reference/*.md` | No |
| Project skills | `skills/` (top-level, outside .claude) | No |
| Project scripts | `scripts/*_tool.py`, `scripts/deploy_*`, `scripts/health_*` | No |
| Project app | `jarvis-app/`, `primitives/`, `stores/` | No |

### Fixed: CLAUDE.md Strategy (Provisional)
CLAUDE.md is excluded from v1 sync. It currently mixes framework rules and project config. Resolution: framework operating rules move to `.claude/rules/` files (which DO sync); CLAUDE.md becomes a project-specific file that references the rules. BuildOS provides a `CLAUDE.md` template in `templates/`.

### Fixed: MVP Scope
v1 is manual/on-demand sync only:
- No session-close hooks, no CI automation, no PR bots
- No consumer update mechanism redesign
- No Copier or template tooling
- Automation added only after the boundary is proven clean

### Fixed: Go/No-Go Threshold
After the boundary audit: if >30% of framework file lines contain project-specific references, revisit approach (may need a more gradual migration or Approach B hybrid). If <30%, proceed.

### Fixed: Sync Script Interface
```
buildos-sync.sh [--dry-run] [--commit] [--push]
  1. Diff framework files (OpenClaw vs claude-build-os)
  2. Show changed file list + line-count diff
  3. Run security scan (banned terms grep) — abort if any match
  4. --dry-run: stop here (default)
  5. --commit: copy files + commit to claude-build-os
  6. --push: commit + push to GitHub
```

### Fixed: Script Certification
Each framework script/hook must pass before sync:
- [ ] No project-local imports or hardcoded paths
- [ ] Dependencies documented (what it needs installed)
- [ ] No proprietary defaults (model names, API endpoints)
- [ ] Tested independently of project-specific code
