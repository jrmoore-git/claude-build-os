# Brief: BuildOS Sync Enforcement

Generated by /define refine on 2026-04-03

## Problem Statement
Framework files drift between openclaw (downstream project) and claude-build-os (upstream generic framework). `buildos-sync.sh` exists but is manual — nothing forces it to run. `/ship` checks it as a SOFT gate (advisory). `deploy_all.sh` doesn't include it. Result: 5 files currently diverged.

## Proposed Approach
Three changes to make sync automatic and enforced:

1. **Add `buildos-sync.sh --push` as final step of `deploy_all.sh`** — every deploy auto-syncs to BuildOS. Banned-terms scan is the genericization guard (already exists and works).
2. **Promote `/ship` BuildOS gate from SOFT to HARD** — can't ship if files diverge and sync fails.
3. **No new hooks or post-commit checks** — the deploy path is the single enforcement point. Simpler than adding hooks.

## Key Assumptions
- Banned-terms scan in `buildos-sync.sh` is sufficient to prevent project-specific content leaking to BuildOS (it catches Justin, Jarvis, OpenClaw, etc.)
- All framework files that should sync are already in the MANIFEST array
- BuildOS repo is always available at `~/claude-build-os`

## Risks
- `deploy_all.sh` fails if BuildOS repo is in a dirty state or has conflicts — need graceful handling
- Network failure on `git push` blocks deploy — should warn but not block entire deploy
- HARD gate in `/ship` could block shipping for unrelated BuildOS issues

## Simplest Testable Version
1. Add sync step to `deploy_all.sh` with error handling
2. Change `/ship` gate from SOFT to HARD
3. Test: make a framework file change, run `deploy_all.sh`, verify BuildOS is updated
