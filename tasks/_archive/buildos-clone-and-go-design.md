## Problem Statement

BuildOS is a generalized framework for building AI-assisted products with Claude Code. Justin uses it daily through OpenClaw/Jarvis and wants anyone (team members, collaborators) to clone the repo and start building immediately with all accumulated learnings — hooks, rules, skills, scripts, governance.

Today, cloning BuildOS gives you the files but not a working system. Hooks reference `/opt/homebrew/bin/python3.11` (macOS Homebrew-specific), there's no setup script, gate hooks will block work without explanation, and the tier system requires understanding before you can be productive. The `/setup` skill is a conversation guide, not automated setup.

## Demand Evidence

- Justin actively shares BuildOS with team members and on GitHub
- Every new product Justin builds starts from BuildOS
- The framework has 26 skills, 17 hooks, 12 scripts — significant accumulated value that's wasted if onboarding is painful
- Current state: new user clones → tries to edit a file → gate hook blocks → confusion → gives up or removes hooks

## Status Quo

Clone → read README → manually install python3.11 → manually install ruff → manually copy .env.example → manually configure LiteLLM → manually figure out why hooks are blocking edits → manually create plan artifacts. ~30-60 minutes of friction before writing a single line of code.

## Target User & Narrowest Wedge

**Target:** Engineer who clones BuildOS to start a new Claude Code project. Wants governance infrastructure without building it from scratch.

**Narrowest wedge:** A `setup.sh` that detects the platform, resolves binary paths, installs git hooks, validates dependencies, and generates a working `.claude/settings.json` with correct absolute paths. Clone → run setup.sh → start building.

## Premises

1. All hooks must work on macOS (Homebrew) and Linux (apt/dnf) — Windows/WSL is nice-to-have, not required (AGREED — Justin's team is macOS, but Linux support costs almost nothing)
2. Tier 0/1 should work with zero external dependencies (just git + Claude Code) (AGREED — gate hooks should not fire until the user opts into governance)
3. The setup script replaces manual README steps, not the `/setup` skill (AGREED — `/setup` guides project design decisions; `setup.sh` handles infrastructure)
4. Hardcoded paths in hooks are the #1 blocker (AGREED — every other issue has workarounds)

## Approaches Considered

### Approach A: setup.sh + Dynamic Path Resolution (RECOMMENDED)

Single setup script that:
1. Detects platform (macOS/Linux) and package manager
2. Finds or installs python3 (3.11+), ruff
3. Generates `.claude/settings.json` with correct absolute paths for this machine
4. Installs git hooks (pre-commit banned-terms)
5. Copies `.env.example` → `.env` if not exists
6. Creates `docs/`, `tasks/`, `stores/` directories from templates
7. Runs a preflight check (verify all dependencies, report what works)
8. Asks for governance tier (0-3) and configures accordingly

**Key change:** `.claude/settings.json` becomes a **generated file** (in .gitignore), not a committed file. A template (`settings.json.template`) is committed with `{{PYTHON3}}` and `{{PROJECT_ROOT}}` placeholders. `setup.sh` renders it.

Effort: M | Risk: Low
Pros: Zero-to-working in one command, platform-portable, no new dependencies, backward-compatible
Cons: Must maintain setup.sh as hooks change, generated settings.json means manual hook edits need re-rendering

### Approach B: Makefile + Docker Dev Container

Docker container with all dependencies pre-installed. `make setup` bootstraps everything.

Effort: L | Risk: Med
Pros: Perfectly reproducible, no platform issues
Cons: Requires Docker (heavy dependency), Claude Code doesn't run in containers well, adds complexity that contradicts BuildOS philosophy

### Approach C: Copier Template Engine

Use Copier to generate project from BuildOS template with variable substitution.

Effort: L | Risk: Med
Pros: Industry-standard templating, handles updates
Cons: New dependency (Copier), learning curve, reverses natural workflow, already rejected in prior design (buildos-upstream-propagation)

## Recommended Approach

**Approach A: setup.sh + Dynamic Path Resolution.**

Rationale:
1. **Matches the philosophy** — BuildOS is about simplicity and disk-based state. A shell script is the simplest automation.
2. **Solves the real problem** — hardcoded paths and missing setup are 90% of the friction. This fixes both.
3. **No new dependencies** — just bash. Works on any Unix system.
4. **Backward-compatible** — existing users can ignore setup.sh and keep their manual config.
5. **Template approach for settings.json** — the right abstraction. Settings.json has absolute paths that MUST be machine-specific. Making it generated (like .env) is correct.

## Implementation Plan

### 1. settings.json template (`settings.json.template`)
- Replace all absolute paths with `{{PYTHON3}}`, `{{PROJECT_ROOT}}`
- Commit template, gitignore the generated file
- setup.sh renders it via sed/envsubst

### 2. Hook portability
- Replace `/opt/homebrew/bin/python3.11` → `{{PYTHON3}}` in template
- Replace `/opt/homebrew/bin/ruff` → `{{RUFF}}` (or remove if not found)
- Python script shebangs: change `#!/usr/bin/env python3.11` → `#!/usr/bin/env python3` (3.11 is not universal; python3 + version check at runtime is more portable)

### 3. setup.sh structure
```
setup.sh
├── detect_platform()     # macOS/Linux, package manager
├── find_python()         # locate python3.11+ (try python3.11, python3, python)
├── find_ruff()           # locate ruff (optional, warn if missing)
├── render_settings()     # template → .claude/settings.json
├── install_git_hooks()   # symlink pre-commit
├── init_project_files()  # copy templates → docs/, tasks/
├── select_tier()         # ask governance level (0-3)
├── configure_tier()      # enable/disable hooks based on tier
├── preflight_check()     # verify everything works, report status
└── print_next_steps()    # what to do now
```

### 4. All hooks enabled by default
No tier selection. The full governance stack is the product. setup.sh enables everything — if someone cloned this repo, they want the battle-tested standard.

## Open Questions

1. **Should setup.sh be idempotent?** Yes — running it twice should detect existing config and offer to update or skip. This is important for when hooks change upstream.
2. **Git hooks: symlink or copy?** Symlink (so updates to hooks/ are picked up automatically). The pre-commit-banned-terms.sh already uses this pattern.
3. **Should the generated settings.json be in .gitignore?** Yes — it's machine-specific, like .env. But this means new contributors must run setup.sh. That's acceptable — it's a one-time cost.

## Success Criteria

1. `git clone && ./setup.sh` produces a working BuildOS project in <60 seconds
2. All hooks work on macOS (Homebrew) and Linux (apt) without manual path editing
3. Tier 0 users can start immediately with zero dependencies beyond git
4. Tier 2 users get a clear checklist of what they need (LiteLLM, API keys)
5. Running setup.sh on an existing project is safe (idempotent)
