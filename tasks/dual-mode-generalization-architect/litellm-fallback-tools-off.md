# litellm-fallback-tools-off — Challenger Reviews

## Challenger A (architect) — Challenges
## Challenges

1. [ASSUMPTION] [MATERIAL] [COST:TRIVIAL]: The proposal assumes `ANTHROPIC_API_KEY` is in the environment because "Claude Code already has configured" it. This is not reliably true — Claude Code users on subscription auth (OAuth/Max plan) often do *not* have `ANTHROPIC_API_KEY` set; the key is managed internally. Fallback must gracefully handle missing key (clear error pointing to how to get one) rather than crashing, otherwise the "zero additional setup" promise breaks for a significant user segment. SPECULATIVE on segment size, but the failure mode is certain when the key is absent.

2. [RISK] [MATERIAL] [COST:TRIVIAL]: Silent correlated-blind-spot risk. A one-line stderr message is easy to miss, and downstream artifacts have "identical structure whether generated by 3 models or 1." Hooks/gates consuming these artifacts cannot distinguish review quality. Recommend adding a `review_mode: single_model | multi_model` field to artifact frontmatter so gates can optionally require multi-model for high-stakes decisions. This preserves the "downstream doesn't need to know" default while not foreclosing the option.

3. [UNDER-ENGINEERED] [ADVISORY]: No mention of caching the fallback detection. If LiteLLM is down and `debate.py` makes N persona calls, does each one re-attempt localhost:4000 and eat the connection-refused timeout? Should be a one-shot probe per process invocation.

4. [RISK] [ADVISORY]: `refine` command's model rotation becoming single-model iterative loop is a meaningful semantic change — iterating the same model on its own output risks sycophantic convergence rather than genuine refinement. Worth considering whether `refine` should refuse to run in fallback mode, or cap at fewer rounds, rather than silently degrading.

5. [ALTERNATIVE] [ADVISORY]: Rather than hand-rolling `urllib.request` against Anthropic's Messages API, consider whether the Anthropic Python SDK (likely already a transitive dep via Claude Code tooling) is acceptable. Hand-rolled HTTP against a versioned API is a small maintenance tax. If dependency minimalism is a hard constraint, urllib is fine — but name that constraint explicitly.

6. [ASSUMPTION] [ADVISORY]: "Connection refused on localhost:4000" as the detection signal is narrow. LiteLLM can also fail via: proxy up but misconfigured (returns 500s), proxy up but one provider key invalid (partial failure), slow/hanging proxy. Define whether those count as "unavailable" and trigger fallback, or surface as errors. Current proposal only addresses the cleanest failure mode.

## Concessions

1. The core insight is correct: the delta between "no review" and "single-model structured review through 3 lenses" is larger than the delta between single-model and multi-model. Prioritizing onboarding over review quality ceiling is the right call.
2. Preserving artifact format so hooks/skills are oblivious is the right abstraction boundary — the fallback is genuinely transparent to downstream consumers.
3. Scope is tight and well-bounded: two files, no config changes, no new abstractions.

## Verdict

REVIEW — APPROVE with build conditions. The design is sound; the MATERIAL findings (missing API key handling, artifact mode labeling) are both TRIVIAL fixes that should land with the change rather than block it. Challenges 3–6 are judgment calls the author can resolve during implementation.
